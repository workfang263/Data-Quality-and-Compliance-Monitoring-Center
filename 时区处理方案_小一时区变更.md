# 时区处理方案：负责人"小一"时区从UTC+8变为UTC-8

## 📋 问题描述

负责人"小一"的Shoplazza店铺和Facebook广告账户时区都从UTC+8变为UTC-8，需要：
1. 正确识别并处理UTC-8时区的数据
2. 统一转换为北京时间（UTC+8）存储到数据库
3. 不影响其他负责人（仍为UTC+8）的数据处理

## 🎯 解决方案

### 方案概述
1. **创建时区配置表**：存储每个负责人的时区信息
2. **修改数据同步脚本**：根据负责人查询时区配置，进行时区转换
3. **统一存储**：所有数据统一转换为北京时间（UTC+8）存储

## 📊 数据库设计

### 1. 创建时区配置表（支持多层级配置）

**设计思路：**
- **负责人级别**：适用于该负责人下的所有店铺和广告账户（默认）
- **店铺级别**：可以覆盖负责人配置（如果某个店铺时区不同）
- **广告账户级别**：可以覆盖负责人配置（如果某个账户时区不同）
- **优先级**：店铺/账户配置 > 负责人配置 > 默认UTC+8

```sql
-- 1. 负责人时区配置表（默认配置）
CREATE TABLE IF NOT EXISTS owner_timezone_mapping (
  id INT PRIMARY KEY AUTO_INCREMENT,
  owner VARCHAR(255) NOT NULL UNIQUE,
  timezone VARCHAR(64) NOT NULL COMMENT '时区名称，如 Asia/Shanghai, America/Los_Angeles',
  timezone_offset DECIMAL(4,1) NOT NULL COMMENT '时区偏移量（小时），如 8.0 表示 UTC+8, -8.0 表示 UTC-8',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 2. 店铺时区配置表（可选，覆盖负责人配置）
CREATE TABLE IF NOT EXISTS store_timezone_mapping (
  id INT PRIMARY KEY AUTO_INCREMENT,
  shop_domain VARCHAR(255) NOT NULL UNIQUE,
  timezone VARCHAR(64) NOT NULL COMMENT '时区名称',
  timezone_offset DECIMAL(4,1) NOT NULL COMMENT '时区偏移量（小时）',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 3. 广告账户时区配置表（可选，覆盖负责人配置）
CREATE TABLE IF NOT EXISTS ad_account_timezone_mapping (
  id INT PRIMARY KEY AUTO_INCREMENT,
  ad_account_id VARCHAR(64) NOT NULL UNIQUE,
  timezone VARCHAR(64) NOT NULL COMMENT '时区名称',
  timezone_offset DECIMAL(4,1) NOT NULL COMMENT '时区偏移量（小时）',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 初始化：所有负责人默认UTC+8
INSERT INTO owner_timezone_mapping (owner, timezone, timezone_offset) 
SELECT DISTINCT owner, 'Asia/Shanghai', 8.0
FROM store_owner_mapping
WHERE owner IS NOT NULL
ON DUPLICATE KEY UPDATE timezone=VALUES(timezone), timezone_offset=VALUES(timezone_offset);

-- 设置"小一"的时区为UTC-8（负责人级别）
INSERT INTO owner_timezone_mapping (owner, timezone, timezone_offset) 
VALUES ('小一', 'America/Los_Angeles', -8.0)
ON DUPLICATE KEY UPDATE timezone=VALUES(timezone), timezone_offset=VALUES(timezone_offset);
```

### 2. 时区查询优先级说明

**查询逻辑（从高到低优先级）：**
1. **店铺/账户级别配置**：如果存在，直接使用
2. **负责人级别配置**：如果店铺/账户没有配置，使用负责人的配置
3. **默认配置**：如果都没有配置，使用UTC+8（Asia/Shanghai）

**优势：**
- ✅ **灵活**：支持不同粒度的配置
- ✅ **方便**：大多数情况下只需配置负责人级别
- ✅ **可扩展**：如果某个店铺或账户时区特殊，可以单独配置
- ✅ **向后兼容**：没有配置的自动使用默认值

### 1. 修改 `data_sync.py` - Shoplazza店铺数据同步

**需要修改的地方：**

#### 1.1 添加时区查询函数（支持多层级配置）

```python
def get_timezone_config(conn, shop_domain: str = None, ad_account_id: str = None, owner: str = None) -> Dict[str, Any]:
    """
    获取时区配置（支持多层级优先级）
    
    优先级：店铺/账户配置 > 负责人配置 > 默认UTC+8
    
    Args:
        conn: 数据库连接
        shop_domain: 店铺域名（可选，用于店铺数据同步）
        ad_account_id: 广告账户ID（可选，用于广告数据同步）
        owner: 负责人名称（可选，用于回退到负责人配置）
    
    Returns:
        {'timezone': 'Asia/Shanghai', 'timezone_offset': 8.0}
    """
    with conn.cursor() as cur:
        # 优先级1：店铺级别配置
        if shop_domain:
            cur.execute("""
                SELECT timezone, timezone_offset 
                FROM store_timezone_mapping 
                WHERE shop_domain = %s
            """, (shop_domain,))
            result = cur.fetchone()
            if result:
                return {
                    'timezone': result['timezone'],
                    'timezone_offset': float(result['timezone_offset'])
                }
        
        # 优先级2：广告账户级别配置
        if ad_account_id:
            cur.execute("""
                SELECT timezone, timezone_offset 
                FROM ad_account_timezone_mapping 
                WHERE ad_account_id = %s
            """, (ad_account_id,))
            result = cur.fetchone()
            if result:
                return {
                    'timezone': result['timezone'],
                    'timezone_offset': float(result['timezone_offset'])
                }
        
        # 优先级3：负责人级别配置
        if owner:
            cur.execute("""
                SELECT timezone, timezone_offset 
                FROM owner_timezone_mapping 
                WHERE owner = %s
            """, (owner,))
            result = cur.fetchone()
            if result:
                return {
                    'timezone': result['timezone'],
                    'timezone_offset': float(result['timezone_offset'])
                }
        
        # 优先级4：默认UTC+8
        return {'timezone': 'Asia/Shanghai', 'timezone_offset': 8.0}


# 为了向后兼容，保留旧的函数名（内部调用新函数）
def get_owner_timezone(conn, owner: str) -> Dict[str, Any]:
    """获取负责人的时区配置（向后兼容）"""
    return get_timezone_config(conn, owner=owner)
```

        # 如果 timezone_offset == 8.0，已经是北京时间，不需要转换

## ⚠️ 注意事项

### 1. 时区转换计算

- **UTC-8 → UTC+8**：需要加 16 小时
- **UTC+8 → UTC+8**：不需要转换（已经是北京时间）

计算公式：
```python
hours_to_add = 8.0 - timezone_offset
# 例如：8.0 - (-8.0) = 16.0
# 例如：8.0 - 8.0 = 0.0（不需要转换）
```

### 2. 日期边界处理

- Facebook返回的 `date_start` 是账户时区的日期
- 转换到北京时间后，日期可能会变化（跨天）
- 需要确保过滤逻辑使用转换后的北京时间

### 3. 数据一致性

- 所有数据统一存储为北京时间（UTC+8）
- 前端显示时不需要再做时区转换
- 聚合逻辑使用 `DATE(time_hour)` 按北京时间日期聚合

### 4. 扩展性

- 如果未来有其他负责人也需要不同的时区，只需在 `owner_timezone_mapping` 表中添加配置
- 代码会自动根据配置进行时区转换

## 📌 总结

### 方案优势

通过创建多层级时区配置表和修改数据同步脚本，可以实现：

1. ✅ **正确识别和处理不同时区的数据**
2. ✅ **统一转换为北京时间存储**
3. ✅ **不影响其他负责人的数据处理**
4. ✅ **支持未来扩展**：
   - **负责人级别**：大多数情况下，只需配置负责人时区，该负责人下的所有店铺和账户都会使用
   - **店铺级别**：如果某个店铺时区特殊，可以单独配置（覆盖负责人配置）
   - **账户级别**：如果某个广告账户时区特殊，可以单独配置（覆盖负责人配置）
5. ✅ **配置灵活**：支持多层级配置，优先级清晰
6. ✅ **向后兼容**：没有配置的自动使用默认值（UTC+8）
