# 店铺运营全链路收口方案

## 目标

解决当前店铺运营子系统“配置已数据库化，但报表/同步仍部分依赖旧常量”的半升级状态，打通以下链路：

- 新增店铺后，能在报表页出现，即使暂时无数据也能显示空区块
- 新增店铺后，默认订单同步和花费同步都能纳入该店
- 店铺 token 来源与映射编辑页保持单一事实来源，避免双维护
- 三家及以上店铺的数据按 `shop_domain` 严格隔离，互不干扰
- Facebook campaign 花费同步进入日常自动运维，支持历史回溯与稳定补数
- 配置中心交互收口：折叠逻辑统一、默认展开/收起符合业务习惯

### 1. 新增店铺在配置中心可见，但报表页不显示

**根因**：报表接口仍使用旧常量 `STORE_OPS_SHOP_DOMAINS` 作为店铺范围，而不是读取 `store_ops_shop_whitelist`。

### 2. 映射编辑页已经新增 token，但店铺运营同步不认

**根因**：映射页新增店铺时已把 `access_token` 写入主系统表 `shoplazza_stores`，但店铺运营订单同步仍从 `store_ops_constants.py` 的 `_ENV_TOKEN_BY_SHOP` 读取固定两家 env token。

### 3. 新增店铺即使绑定广告账户，也可能无法进入完整报表链路

**根因**：

- campaign 花费同步 `fb_campaign_spend_sync.py` 已经从 `store_ops_shop_ad_whitelist` 读启用账户，属于数据库驱动；

- 报表页即使放开店铺范围，如果订单同步未纳入新店，也只会出现空卡片或只有花费。

## 风险与对应解决方法

- **风险：放开白名单后，新增店铺立即出现在报表，但无订单/花费看起来像“空店”**
  - 解决：接受空卡片为正常过渡状态；用 header/空态提示说明“暂无数据”
- **风险：token 来源切换时影响现有两店稳定性**
  - 解决：先做 DB 优先 + env fallback，确认稳定后再删 fallback
- **风险：报表和同步分别手写 SQL，后续再次口径分叉**
  - 解决：强制把白名单与 token 读取都下沉到 `database_new.py`，由上层统一复用
- **风险：Windows 任务计划跑 bat 时路径/权限出错**
  - 解决：使用 `cmd /c` + 项目根目录作为“起始于”，并先手工跑通 bat
- **风险：campaign 对账 warning 数量较高引起误判**
  - 解决：继续将 warning 视作异常发现，不阻断任务；后续按账号/日期专项排查 `hourly_sum=0` 的来源

## 实施顺序建议

1. 先改报表范围（最短路径解决“店铺 3 不显示”）
2. 再改默认订单同步范围（让新店真正进订单主链路）
3. 再统一 token 来源（解决新增店铺可扩展接入问题）
4. 最后再做运维化和任务计划固化（让花费同步稳定进入日常）

## 关键结论

这次改造的关键不是重做数据模型，而是把已经存在的数据库白名单和主系统 token 事实来源真正接入报表与同步入口。当前问题的本质是“入口名单没有切过去”，不是“配置失败”或“隔离模型错误”。
