# 店铺运营页：el-table 列排序（唯一执行方案 · 防递归版）

**文档地位**：本文件为**唯一**执行规格；实现只按本章执行，文内不拆备选路径 A/B。

**与当前代码关系**：[frontend/src/views/StoreOps.vue](../../frontend/src/views/StoreOps.vue) 现为 `:data="shop.employee_rows"`、无排序；本方案在**保留现有布局与列（头像、FB、ROAS 等）**前提下接入排序。

---

## 1. 前车之鉴（根因）

曾出现的 `Maximum recursive updates exceeded` 来自闭环：`report` 更新 → `watch` 内 `table.sort()` → Element Plus 派发 `sort-change` → `onSortChange` 写 `sortState` → `:data` 若每次返回**新数组引用** → 表格重渲染 → 再次进入 `watch`/`sort`。

**绝对禁止的反模式组合**：

- `watch(report, { deep: true })`（本项目 `loadReport` 为整对象替换，**只浅监听 `report.value`**）。
- **无锁**时对多表 `table.sort()`，且 `onSortChange` **不区分**程序化事件与用户点击。
- 父组件模板 **`:data="sortedEmployeeRows(shop)"`** 等**每次渲染都执行函数并返回新数组**的写法。

**斩断方式**：子组件内 **`computed` 稳定依赖** + **`isSyncingHeader` 锁**拦截程序化 `sort-change` + **浅 `watch` + 同步代数** + **解锁前固定 50ms**（见下文）。

## 2. 产品规则（Slug 仅展示）

- **可排序列（6）**：`direct_sales`、`allocated_from_public_pool`、`total_sales`、`fb_spend`、`roas`、`direct_order_count`；`sortable="custom"`。
- **员工 Slug 列**：`sortable=false`；按指标排序时**整行**重排，Slug 与六列同行，无需单独排序。
- **默认**：按 `shop_domain`；`sortStateByShop` 无记录时 `getSortState(domain)` 返回 `{ prop: 'direct_sales', order: 'descending' }`（懒默认）。
- **`loadReport` / 换日期**：只更新 `report`，**不清空** `sortStateByShop`。
- **ROAS**：`null`/`undefined` 用 `normalizeRoasForSort` → `Number.NEGATIVE_INFINITY`。
- **平局**：`employee_slug` 字符串 `localeCompare` 升序为第二关键字。

## 3. 架构（唯一实现策略）

### 3.1 父组件 StoreOps.vue

- **状态**：`sortStateByShop: Ref<Record<string, { prop: StoreOpsSortProp; order: 'ascending' | 'descending' }>>`；`SORT_PROPS` + `isSortProp` 收窄类型。
- **`onSortChange`**：① `isSyncingHeader` 为 true → return；② 校验/回退默认；③ 与当前 `getSortState(domain)` 相同则 return；④ `sortStateByShop.value = { ...sortStateByShop.value, [domain]: next }`。

### 3.2 子组件 StoreOpsShopEmployeeTable.vue（必选）

- **Props**：`shop`（或 `employee_rows` + `shop_domain`）、**`sortProp`、`sortOrder` 两个标量**（父模板用 `getSortState(shop.shop_domain).prop` / `.order`）。禁止父组件每次传入新对象 `{ prop, order }` 作为单 prop。
- **`:data`**：绑定内部 **`computed sortedRows`**；禁止父级函数式 `:data`。
- **`:default-sort`**：在子组件 `<el-table>` 上通过 **`computed`（如 `defaultSortConfig`）** 绑定 **`:default-sort="defaultSortConfig"`**，**禁止**在模板内联 **`:default-sort="{ prop: sortProp, order: sortOrder }"`**（每次渲染新对象引用会触发 Element Plus 表头反复同步，首屏即可 `Maximum recursive updates exceeded`）。Element Plus 仅在**初次挂载**时用其画表头箭头，与「行序由 `sortedRows` 决定」互补：**首屏**不必依赖 `watch` 再 `applySort` 才有箭头；**后续**换日期等仍靠父级 `watch` + `applySort` 对齐。若仅 `sortProp`/`sortOrder` 变化而子组件未因 `:key` 重挂，`default-sort` 不会自动更新，属已知限制，以 `applySort` 为准。
- **Props 与响应式（强制）**：**禁止**在 `<script setup>` 顶层对 props 做 **`const { sortProp, sortOrder } = defineProps(...)`** 这类解构（会丢失响应式）。**必须**在 `computed` / `watch` 内使用 **`props.sortProp` / `props.sortOrder`**，或对 `defineProps` 结果使用 **`toRefs(props)`** 再解构。示例：`const sortedRows = computed(() => { const p = props.sortProp; const o = props.sortOrder; ... })`。
- **`@sort-change`**：`emit('sort-change', payload)`，父组件转 `onSortChange(domain, payload)`。
- **排序逻辑**：`compareRows` 等可置于 [frontend/src/utils/storeOpsSort.ts](../../frontend/src/utils/storeOpsSort.ts) 共用。
- **`sortedRows` 空值兜底（强制）**：`employee_rows` 可能缺省或为 `null`（接口异常、某店无员工）。`computed` 内第一步使用 **`const rows = props.shop?.employee_rows ?? []`**，禁止对 `undefined` 做 `[...rows]`。`sortProp`/`sortOrder` 由父级 `getSortState` 保证有值时仍可直接参与排序；若实现上存在空串等边界，可先判断再 `return rows` 避免白屏。

### 3.3 表头同步（浅 watch + 锁 + 代数 + try/finally）

- **`tableRefByShop`**：存子组件实例（见第 4.1 节 **applySort**）。
- **`isSyncingHeader`**、**`headerSyncGeneration`**：每轮有效同步开始 `const gen = ++headerSyncGeneration.value`。
- **`watch(() => report.value, async ...)`**：禁止 `deep: true`。若 `!newReport?.shops` 则尽早 return（不上锁）。
- **锁的释放（极度重要）**：上锁后，**`try { ... } finally { ... }` 包裹**从 `nextTick`、遍历 `applySort`、`await setTimeout(50)` 在内的异步流程，避免中间任一步 **throw**（如某店数据异常、子组件未就绪）导致 **`isSyncingHeader` 永不复位**、全页排序交互永久死锁。
- **`finally` 解锁条件**：仅当 **`gen === headerSyncGeneration.value`** 时执行 **`isSyncingHeader.value = false`**（与旧逻辑一致：更新过程中若又触发新一轮报表，旧轮不解锁，由新一轮收尾）。
- **遍历域名**：以 **`newReport.shops`** 为准逐项取 `domain`，再 **`tableRefByShop` 中对应实例**调用 `applySort`；不要仅依赖 `Object.entries(tableRefByShop)`（避免残留已下线店铺的 key）。
- **`{ flush: 'post' }`（禁止删除）**：`watch` 第三个参数**必须**保留 **`{ flush: 'post' }`**。默认 `pre` 时回调可能在**子组件尚未挂载、函数式 ref 尚未写入**之前执行，新店首次出现会导致 **`applySort` 拿不到实例**。`post` 在 DOM 更新**之后**执行，保证新 `StoreOpsShopEmployeeTable` 已渲染且 ref 已收集。
- **典型骨架**：
```ts
watch(
  () => report.value,
  async (newReport) => {
    if (!newReport?.shops?.length) return
    const gen = ++headerSyncGeneration.value
    isSyncingHeader.value = true
    try {
      await nextTick()
      for (const shop of newReport.shops) {
        const domain = shop.shop_domain
        const st = getSortState(domain)
        tableRefByShop.value[domain]?.applySort(st.prop, st.order)
      }
      await nextTick()
      await new Promise((r) => setTimeout(r, 50))
    } finally {
      if (gen === headerSyncGeneration.value) {
        isSyncingHeader.value = false
      }
    }
  },
  { flush: 'post' },
)
```

## 4. 开发避坑（落地规范）

### 4.1 子组件 `defineExpose`：必须方法透传（极其重要）

父组件通过**子组件封装**无法直接拿到内部 `el-table` 实例调用 `.sort()`。子组件内：

- 类型导入（避免 TS 报未定义）：**`import type { TableInstance } from 'element-plus'`**。
- 使用 `const innerTableRef = ref<TableInstance | null>(null)` 绑定在 **`<el-table ref="innerTableRef">`**。
- **`defineExpose({ applySort: (prop, order) => { innerTableRef.value?.sort(prop, order) } })`**（或等价命名，全项目统一即可）。

父组件同步表头时**只调用** `instance.applySort(prop, order)`，**禁止** `instance.sort(...)`（除非暴露的就是 TableInstance）。

### 4.2 父组件函数式 `ref`：卸载时必须清理（防泄漏与脏引用）

`v-for` 店铺列表动态变化时，**必须**使用函数式 ref，且在 `el === null` 时删除对应 key：

```vue
<StoreOpsShopEmployeeTable
  v-for="shop in report.shops"
  :key="shop.shop_domain"
  :ref="(el) => registerStoreOpsTableRef(shop.shop_domain, el)"
/>
```
```ts
const tableRefByShop = shallowRef<
  Record<string, InstanceType<typeof StoreOpsShopEmployeeTable> | null>
>({})

function registerStoreOpsTableRef(domain: string, el: unknown) {
  if (!domain) return
  const map = tableRefByShop.value
  if (el) {
    const inst = el as InstanceType<typeof StoreOpsShopEmployeeTable>
    if (map[domain] !== inst) {
      map[domain] = inst
    }
  } else if (map[domain]) {
    delete map[domain]
  }
}
```

说明：存**组件实例**宜用 **`shallowRef`**，避免把实例做成深度响应式。落地时可对 **同一 `shallowRef` 指向的 map 原地赋值**，且 **仅在 `map[domain] !== el `时写入**，减少 `v-for` + 函数式 `ref` 反复回调时的无意义响应式替换与抖动；**务必**在 `el == null` 时 `delete` 对应 key。若仍采用「每次 `{ ...spread }` 替换整个对象」的写法亦可，但需警惕与内联 `ref` 回调叠加时的额外渲染。

### 4.3 解锁延时：计划内**定死 50ms**

- 在 `applySort` 循环与 `nextTick` 之后，解锁前增加：**`await new Promise((r) => setTimeout(r, 50))`**。
- **理由**：Element Plus 的 `sort` 与 `sort-change` 可能落在**宏任务 / 渲染帧**之后；仅 `queueMicrotask` 或 `setTimeout(0)` 仍可能早于 EP 队列，锁提前结束会导致穿透。50ms 为**工程默认**，体感不可辨，且显著降低穿透概率。
- **表述约束**：不得写「100% 保证」；应写「显著降低锁穿透概率的默认策略」。

## 6. PR / 提测标准

- 加载、换日期、多点排序：控制台**无** `Maximum recursive updates exceeded`。
- 首次进入：各店 **direct_sales 降序**，表头与行序一致。
- ROAS 升/降序：`null` 行在最上/最下。
- 某店改排序后换日期：行序与箭头仍一致。
- **抽样**：仅改 A 店排序时，B 店子组件不因 A 的 `sortState` 变化而无效重算（依赖本店 props）。

## 7. 附录：为何「仅升序/降序」仍会牵涉较多工程细节

「排序」本身（比较两行大小）只有几十行逻辑。**复杂度来自环境与约束叠加**，而非比较函数：

1. **Vue 响应式**：模板里随意 `:data="fn(shop)"` 会在每次父级渲染产生**新数组引用**，触发子树反复更新；必须用 **computed/子组件** 控制依赖粒度。
2. **Element Plus 双状态**：表格**行数据**由你控制，**表头箭头/内部排序状态**由组件控制；用 `sortable="custom"` 后，程序化 `sort()` 会再发 **`sort-change`**，若与「改本地状态」形成回路就会 **Maximum recursive updates**；因此需要 **锁 + 拦截**。
3. **多实例**：每店一张表，`ref` 收集、`loadReport` 后对齐箭头、卸载清理，都是**多实例**下的额外规则。
4. **持久化与刷新**：换日期后**行序**要仍按记忆排序，**箭头**也要对齐，才会出现 **`watch` + `applySort`**；这又回到第 2 点的环路风险。
5. **容错**：异步链路中任一步抛错若不复位锁，会造成**永久死锁**；因此需要 **try/finally**。

在**静态 HTML 表格**或**无响应式 UI 库**里自己画箭头，可以只做第 1 条里的比较函数；一旦使用 **Vue + Element Plus + 自定义数据排序 + 多表 + 刷新对齐**，就必须把上述边界写进方案。**文档不承诺「数学意义上绝对」不出现递归**，只通过架构与清单把风险压到可接受。

## 8. 开发 Tips（代码级零摩擦补充）

1. **`flush: 'post'`**：与 §3.3 一致，**不得删除**；理由见该节（新店 ref、DOM 时序）。
2. **`TableInstance`**：子组件 `innerTableRef` 使用 **`import type { TableInstance } from 'element-plus'`**（见 §4.1）。
3. **`employee_rows ?? []`**：子组件 `sortedRows` 内必须兜底，见 §3.2。
4. **Element Plus 第三态（升序 → 降序 → `order === null` 取消）**：§3.1 已将 `order === null` 回退为默认 `direct_sales` + `descending`，**行序**由 `sortedRows` 与状态一致。**注意**：**`watch(report)` 仅在报表引用变化时执行**；用户在同一日期范围内仅通过表头点出「取消排序」时，`report` 可能不变，**不能**依赖 `watch` 去 `applySort` 把表头箭头拉回默认列。若实测表头与行序不一致，应在父组件 **`onSortChange` 完成状态写入后**，**`await nextTick()`**，再对该 `domain` 调用 **`tableRefByShop[domain]?.applySort(getSortState(domain).prop, getSortState(domain).order)`**；若该调用再次触发 `sort-change`，须与程序化同步相同使用 **`isSyncingHeader` 短锁**或依赖「状态未变则 return」的守卫，避免回路（具体以 PR 联调为准）。

**对外表述**：避免「绝对无死循环」「绝对顺滑」等绝对化用语；以 §6 提测与控制台无 `Maximum recursive updates` 为准。
