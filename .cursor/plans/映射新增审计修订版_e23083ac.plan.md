## 依据（有理有据）

### 1. Shoplazza 店铺

- [`shoplazza_stores`](db/schema.sql) **无** `timezone` 列；[`data_sync.py`](data_sync.py) 对订单 `placed_at` 按 API 返回的 UTC/带时区串解析后 **统一到北京时间** 写入 [`shoplazza_store_hourly`](db/schema.sql)。
- 因此 **不需要**、也不应在「新增店铺」表单收集「店铺 IANA」——所谓「自动」在此指 **店匠接口提供时间 + 现有同步规则转北京**，不是再去 Meta 拉店铺时区。

**前端提示文案建议（示意）**：

「店铺侧无需配置时区：订单时间以店匠 API 为准，系统将按现有规则统一为北京时间入库。」

### 2. Facebook 广告账户

- [`fb_spend_sync.py`](fb_spend_sync.py) 已通过 [`get_timezone_config`](timezone_utils.py) + [`convert_to_beijing_time`](timezone_utils.py) 将 **账户本地日历/小时** 转为 **`time_hour`（北京时间 naive）**.
- 配置来源优先级：**`ad_account_timezone_mapping` → `owner_timezone_mapping` → 默认 Asia/Shanghai**（见 [`timezone_utils.py`](timezone_utils.py) 注释与实现）。
- 仓库内已有脚本使用 Graph 字段 **`timezone_name`**（例如 [`get_fb_ad_accounts.py`](get_fb_ad_accounts.py) `fields=...timezone_name`），说明 **Meta 侧可提供账户时区名称**，可用于落库。

**结论**：前端不填时区；**在「新增 FB 映射成功」后（或首 sync 前）由后端调用 Graph**（如 `GET /vXX/act_{id}?fields=timezone_name`，或等价节点），将返回值 **规范化** 后写入 [`ad_account_timezone_mapping`](db/schema.sql)（`timezone` + `timezone_offset`，offset 由 `zoneinfo` 计算）。若 Graph 失败或无字段：**审计记 warning**，并回退负责人/上海（与现逻辑一致）。

**前端提示文案建议**：

「无需填写时区：保存后系统将从 Meta 读取该广告账户的 timezone 并用于将花费数据换算为北京时间。」

### 3. TikTok 广告账户

- 公开文档/社区资料：广告主信息接口可返回 **`timezone` / `display_timezone`**（TikTok 常用为 **平台附录中的时区编码**，未必等价于 IANA）。
- 当前 [`tt_spend_sync.py`](tt_spend_sync.py) **须使用** [`timezone_utils.get_timezone_config`](timezone_utils.py)，对 `stat_time_hour` 按账户偏移 **换算为北京时间** 再写入 [`tt_ad_account_spend_hourly`](db/schema.sql)，与 FB 语义对齐。

**实现注意**：需实现 **TikTok 时区编码 → `timezone_offset`（或 IANA）** 的映射层（查官方 Appendix-Time Zone；无法在运行时 `ZoneInfo(tt_code)` 的编码要查表）。拉取失败时回退负责人/上海并记审计。

**前端提示文案建议**：

「无需填写时区：保存后系统将从 TikTok 读取该广告主账户的时区并用于换算为北京时间。」

## 实施阶段（在原有阶段上替换「时区表单」相关条目）

### 阶段 0

不变：**act_** 迁移、历史 `fb_*` / mapping / timezone 表 ID 统一；备份与回滚说明。

### 阶段 1 — 公共模块

- `normalize_fb_ad_account_id`、域名校验、`redact_for_audit`（**token 永不入审计 payload**）。
- **`fetch_and_upsert_fb_ad_timezone(conn, act_id, token)`**：调 Graph，解析 `timezone_name` → IANA 或别名表 → `timezone_offset` → upsert `ad_account_timezone_mapping`。
- **`fetch_and_upsert_tt_ad_timezone(conn, advertiser_id, access_token)`**：按现有 [`TT_CONFIG`](config.py)（BC token）调用广告主信息接口；**编码映射** → upsert `tt_ad_account_timezone_mapping`。
- 失败策略：不阻塞映射创建；**降级** owner/上海 + `mapping_resource_audit` 记 `warning`。

### 阶段 2 — 审计表

不变：**新建结构化审计表** + `Database.log_mapping_audit`；店铺创建审计**不得**含 access_token。

### 阶段 3 — API

- `POST /api/mappings/stores`：`shop_domain`、`owner`、`access_token`、`is_active`；**无时区字段**。
- `POST /api/mappings/facebook`：`ad_account_id`（规范化 `act_`）、`owner`；成功后 **同步或异步** 调用 `fetch_and_upsert_fb_ad_timezone`（需 `FB_LONG_LIVED_TOKEN` 或后端统一读 env）。
- `POST /api/mappings/tiktok`：`ad_account_id`、`owner`；**首期：轮询 `TT_CONFIG.business_centers` 直至命中广告主**，与 [`tt_spend_sync.py`](tt_spend_sync.py) 一致。
- `GET /api/mappings/owners/suggestions`：`q` / `limit`，三表去重联想，**需 `can_edit_mappings`**（与 POST 新增一致）。
- `GET /api/audit/mapping-resources`：审计列表（admin 或映射编辑权限）。
- 现有 PUT 更新负责人：保留；**可选**在 FB PUT 成功后若无时区行则补拉 Meta。

### 阶段 4 — 前端

- [`Mappings.vue`](frontend/src/views/Mappings.vue)：三列「新增」弹窗 **无时区控件**；三处 helper 文案按上文 **分平台区分**；负责人 Autocomplete **优先调用** `owners/suggestions`，失败回退页内去重。
- 审计列表页：[`MappingAuditLog.vue`](frontend/src/views/MappingAuditLog.vue)。

### 阶段 5 — TikTok 同步脚本改造

- 在 [`tt_spend_sync.py`](tt_spend_sync.py) 中：对每个 `advertiser_id` 调用 `get_timezone_config(..., platform="tiktok")`，对解析出的 naive 小时调用 **`convert_to_beijing_time`**，过滤「本地历日 = 请求日」后再 upsert。
- **回归注意**：历史已入库数据若当时假设为北京，重跑需运营确认 **按日期范围回填**。

### 阶段 6 — 测试

- 见 [`tests/test_mapping_audit_timezone_stage6.py`](tests/test_mapping_audit_timezone_stage6.py)：FB/TT mock、脱敏、`convert_to_beijing_time` 与 TT 管线断言。

## 风险与开放项（需在开发中收口）

- **Meta `timezone_name`**：少数值可能不是标准 IANA，需 **别名表**（可迭代维护）。
- **TikTok 时区为附录编码**：须 **编码→offset** 映射（见 `mapping_resource_utils` 内字典与 UTC 串解析）。
- **BC token 与广告主归属**：新增 TT 映射时 **轮询 BC token**，与现有 [`tt_spend_sync`](tt_spend_sync.py) 一致。
