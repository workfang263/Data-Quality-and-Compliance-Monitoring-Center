## 一、项目目标（摘要）

在 Shoplazza 数据看板既有「店铺 / Facebook / TikTok → 负责人」映射能力上，按修订方案落实：

- **体验**：三类「新增映射」**均不填写时区**；文案按平台区分（店匠走订单 API 转北京 vs 广告侧由服务端拉取账户时区）。
- **数据**：FB 广告账户 ID 统一为 **`act_{数字}`**；Meta `timezone_name`、TikTok 广告主时区写入对应 **`*_timezone_mapping` 表**；**审计**结构化落库且 **payload 脱敏（无 token 明文）**。
- **一致性与可维护性**：[`tt_spend_sync.py`](../tt_spend_sync.py) 与 FB 花费用同一套 [`timezone_utils`](../timezone_utils.py)（`get_timezone_config` + `convert_to_beijing_time`），小时花费以**北京时间 naive** 入库。
- **收尾**：补充 **`GET /api/mappings/owners/suggestions`**（三表负责人去重联想），及 **`tests/test_mapping_audit_timezone_stage6.py`** 回归用例。

**刻意不做**：POST 映射后不自动触发全量广告同步（Phase2）；历史 TT 小时表是否回填由业务另定日期范围。

## 三、问题与解决方法

### 1. Swagger 调用映射/审计接口返回 401

- **现象**：未携带 JWT。
- **解决**：在 Swagger **Authorize** 中填入 `Bearer <token>`；非服务未启动问题。

### 2. TikTok 测试数据中负责人被误改

- **现象**：测试账户 owner 与生产认知不一致。
- **解决**：结合历史 `tt_ad_account_spend_hourly` 等表中的 `owner` 字段推断并手工修正映射行；测试数据使用后及时清理。

### 3. 审计 `action` 写 `create` 与实际 upsert 语义不完全一致

- **现象**：资源已存在时仍为「创建」类审计动作。
- **处理**：接受为**语义略粗**，不阻塞上线；若以后要区分，可加 `upsert` 或依据是否新建行写入不同 `action`。

### 4. 负责人联想仅依赖当前页数据

- **现象**：早期 `Mappings.vue` 从已加载三列列表去重生成 Autocomplete，跨页或大数据量时体验不足。
- **解决**：实现 **`GET /api/mappings/owners/suggestions`**（`Database.suggest_mapping_owners` 三表 `UNION` + `LIKE`）；前端请求失败时**保留原页内去重**作为回退。

### 5. 历史 TikTok 小时花费是否重算

- **现象**：旧逻辑若把报表本地时间当作北京时间入库，与现公式不一致。
- **处理**：**不自动回填**；由运营/业务确定日期范围后再批处理或按日重跑 `tt_spend_sync.py`，避免静默改数引发对账争议。
