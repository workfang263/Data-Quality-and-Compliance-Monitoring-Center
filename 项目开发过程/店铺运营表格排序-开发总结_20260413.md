## 一、背景与目标

在 **店铺运营概览**（`StoreOps.vue`）中，每家店铺一张员工指标表。业务需要支持 **6 个指标列**（直接销售额、公共池分摊、合计销售额、广告花费、ROAS、直接订单数）的 **自定义排序**，员工 Slug 列仅展示、不参与排序；**两店分开展示**，排序状态 **按 `shop_domain` 分别记忆**；换日期加载新报表时 **不丢失** 各店已选排序。

技术约束来自 **Vue 3 + Element Plus**：表格 **行数据** 由前端排序，表头 **箭头状态** 由组件内部维护；程序化调用 `table.sort()` 会再次触发 `sort-change`，若与响应式状态形成闭环，易出现 **`Maximum recursive updates exceeded`**。本阶段目标是在不改变后端与 `storeOps` API 字段契约的前提下，完成前端实现并通过类型检查与手测。

## 三、本次遇到的问题与解决方法

### 1. 方案 A：防「报表刷新 ↔ sort-change」闭环（规格 §3）

**现象**：历史上曾出现 `report` 更新 → `watch` 内 `applySort` → `sort-change` → 改状态 → `:data` 不稳定引用 → 再次触发。

**解决**（已写入方案并落地）：

- 子组件用 **`computed sortedRows`** 排序，避免父模板每次渲染 `fn(shop)` 返回新数组。
- 父组件 **`watch(() => report.value, …, { flush: 'post' })`**，内层 **`isSyncingHeader` + `headerSyncGeneration`**，**`try/finally`** 解锁；程序化同步时 **`onSortChange` 若锁为真则直接 return**。
- 换日期后对每店调用子组件暴露的 **`applySort`**，与 **50ms** 延时配合，降低锁穿透概率（工程策略，非数学证明）。

### 2. 联调问题：首屏仍报 `Maximum recursive updates exceeded`

**现象**：页面加载时控制台报错，堆栈指向 **`ElTableHeader` / `ElTable` 的 `mounted`**，表格区域无法正常展示；此时 **`watch(report)` 上的锁尚未单独覆盖「纯首屏渲染」路径**。

**原因**：子组件模板曾使用 **内联** `:default-sort="{ prop: sortProp, order: sortOrder }"`。在 Vue 中，**每次渲染都会生成新的对象字面量（新引用）**；Element Plus 表头对 `default-sort` 的同步会认为「配置又变了」，触发内部更新 → 再次渲染 → 再次新对象 → **与首屏 mounted 叠加形成短回路**，触发 Vue 更新次数上限。

**解决**：

1. **稳定引用**：增加 **`computed`**（如 `defaultSortConfig`），仅在 `sortProp` / `sortOrder` 变化时产生新对象；绑定 **`:default-sort="defaultSortConfig"`**。
2. **减轻 ref 抖动**：父组件 **`registerStoreOpsTableRef`** 改为在 **同一 `shallowRef` 的 map 上原地赋值**，且 **仅当 `map[domain] !== el` 时写入**，卸载时 **`delete`**，避免无意义的整对象替换与反复回调放大更新。

修复后首屏可正常渲染，控制台无递归报错（以实际联调为准）。

## 四、项目开发过程（本仓库上下文）

本仓库为 **Shoplazza 数据看板 / 折线图与运营分析** 类项目（「line chart」工作区），技术栈为 **FastAPI + PyMySQL + Vue 3**。店铺运营能力在既有 **店匠同步、映射、权限** 体系上迭代：后端提供报表与同步 API，前端在统一壳（`PageShell`）下提供 **店铺运营概览** 页。

本次工作属于 **店铺运营前端体验增强**：在已有两店卡片、公共池汇总、员工行（含 FB 花费与 ROAS 等）不变的前提下，接入 **可记忆的多表排序**，并处理 **Vue + Element Plus** 下的响应式与组件边界问题。方案以 `.cursor/plans` 中的单文件规格为唯一执行依据，避免多套架构并行。
