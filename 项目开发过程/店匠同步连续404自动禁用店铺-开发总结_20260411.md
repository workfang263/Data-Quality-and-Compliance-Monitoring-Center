# 店匠同步：连续 HTTP 404 自动禁用店铺 — 开发总结

**文档日期**：2026-04-11  
**关联代码**：[`shoplazza_api.py`](../shoplazza_api.py)、[`data_sync.py`](../data_sync.py)、[`database.py`](../database.py)（`disable_store`）

---

## 一、背景与目标

### 1. 整体项目中的位置

在 **Shoplazza 多店铺看板**体系中，根目录脚本（`data_sync`、`fill_date_data` 等）会按调度对**所有启用店铺**调用店匠 OpenAPI（订单、数据分析等）。若某店铺域名已失效或店铺在平台侧不可用，接口可能长期返回 **HTTP 404**，在重试策略下仍会消耗时间与日志噪音，拖慢整批同步。

### 2. 本次子目标

| 目标 | 说明 |
|------|------|
| 自动止损 | 在**同一轮** `_make_request` 重试序列中，若**连续两次**收到 **404**，则视为该店铺 OpenAPI 路径不可用 |
| 写库禁用 | 调用已有 **`Database.disable_store(shop_domain, reason=...)`**，将 `shoplazza_stores.is_active` 置为禁用 |
| 覆盖范围 | **订单**与**数据分析**任一链路触发「连续两记 404」即可置位；由 `data_sync` 在适当时机执行禁用 |
| 不做 | 不增加「总开关」；不修改 Windows 任务计划 bat，仅改 Python |

---

## 二、实现要点（交接用）

### 1. API 层（`ShoplazzaAPI`）

- 实例上增加 **`_auto_disable_double_404`**（默认 `False`）。
- **`_make_request`** 内用局部变量 **`consecutive_404`**：若响应为 **404** 则累加；**≥2** 时置位 `_auto_disable_double_404 = True`、打 warning、`return None`。
- 若收到**非 404** 的成功路径，**清零** `consecutive_404` 后 `raise_for_status()` 等；**网络异常**分支清零 `consecutive_404`，保留原有重试与 `_last_api_failure` 逻辑。

### 2. 同步层（`data_sync`）

- 辅助函数 **`_auto_disable_store_if_double_404(api, shop_domain, db)`**：若 `_auto_disable_double_404` 为真则 **`disable_store`** 并返回 `True`。
- **`sync_store_data`**（先分析、后订单）：在 **`get_data_analysis_all_pages`** 与 **`get_orders_all_pages`** 之后**各检查一次**，命中则设置 `result` 并 **return**。
- **`sync_store_data_for_ten_minutes`**（先订单、后分析）：同样在两次分页拉取之后**各检查一次**；函数内 **`db = Database()`** 复用。

### 3. 数据层

- **`database.disable_store`** 已存在：更新 `is_active`、`updated_at`；若已禁用则打日志「已经是禁用状态」。

### 4. 后续同步行为

- **`get_active_stores()`** 仅 `is_active = TRUE`，禁用后**新任务**不再请求该店。
- **`fill_date_data.py`** 等在**单次运行开头**拉一次店铺列表：若运行过程中某店被禁用，**当次**仍可能继续用内存中的旧列表对该店请求（见下文「现象与说明」）。

---

## 三、开发过程与验证

1. **方案约定**：连续 404 的计数范围限定在**单次 `_make_request` 内**（含重试循环），与「跨多个 endpoint 各一次 404」区分，避免误杀。
2. **联调**：将测试店铺 `is_active` 置回 1 后跑 `fill_date_data.py yesterday 8`（或单日同步）；日志出现 `OpenAPI 连续两次返回 HTTP 404` 与 `disable_store`；数据库 `is_active=0`。
3. **SQL 校验**：`SELECT shop_domain, is_active, updated_at FROM shoplazza_stores WHERE shop_domain = '...'`。

---

## 四、遇到的问题与处理

### 1. 同一次补数任务里，同一店铺日志刷屏

- **现象**：`fill_date_data` **只在一开始**调用 `get_active_stores()`，144 个时间段并行/串行仍会带着**已禁用**店铺跑满全程；每次时间段仍会请求该店，触发 404 与「已标记自动禁用」/「已经是禁用状态」重复日志；汇总行出现 **`39/40`** 写入成功。
- **性质**：**预期内的实现边界**（内存列表未随 DB 热更新），不是禁用未生效。
- **处理选项**（未强制改代码）：后续若需减少无效请求，可在每批次或禁用后**重读 `get_active_stores()`** 或从内存列表移除该域名；属性能与复杂度权衡。

### 2. 禁用后「下次是否还请求」

- **结论**：**新启动的同步任务**不会再包含该店（`get_active_stores`）；**同一次长任务**内仍可能继续请求直至结束。

### 3. 与 Loguru 改造的关系

- 无强耦合；排查时可在 **`logs/combined-*.log`** 中按 `shop_domain`、`404`、`禁用` 检索。
