## 一、背景

在店铺运营页新增员工表汇总后，联调中出现两个业务口径问题：

1. 父页额外展示的「店铺总销售额（含公共池）」与表内 `Σ total_sales` 口径重叠，易被误解为重复统计。
2. 汇总行 `ROAS` 原先显示 `—`，业务希望改为可直接反映全店效率的汇总指标。

同时，权限管理页中出现 `Sunelva / Anna / Bertlove` 等“非负责人”选项，需确认是否前端异常。

## 二、本窗口改动（已落地）

### 1) 店铺运营汇总口径

- 保留：父页 **店铺总订单数（含公共池）**  
  公式：`public_pool_order_count + Σ direct_order_count`
- 移除：父页 **店铺总销售额（含公共池）** 展示块（避免与汇总行 `Σ total_sales` 双口径并列造成歧义）
- 调整：表尾汇总 `ROAS` 改为 **加权口径**  
  公式：`Σ total_sales / Σ fb_spend`，当 `Σ fb_spend <= 0` 时显示 `—`

## 三、排障问题与结论

### 问题 A：权限页为何出现“额外负责人”

**现象**：权限管理页的“负责人授权”出现 `Sunelva / Anna / Bertlove`。  
**结论**：并非前端渲染错误，而是后端接口 `/api/permissions/owners` 的数据口径决定。

后端 `get_all_owners()` 会从三张映射表合并去重：

- `store_owner_mapping`
- `ad_account_owner_mapping`
- `tt_ad_account_owner_mapping`

只要某字符串曾作为 `owner` 出现，就会进入权限勾选列表。即“额外负责人”本质是映射数据中的历史值或测试值，而非 UI 随机生成。
