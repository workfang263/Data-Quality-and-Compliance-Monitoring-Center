# Loguru 日志体系改造 — 开发总结

**文档日期**：2026-04-11  
**关联方案**：`.cursor/plans/loguru_日志方案_a7896fa3.plan.md`（已标记**实施完成**）  
**适用范围**：项目根目录 Python 脚本、工具链与 **FastAPI 后端**（`backend/`）共用一套落盘与控制台策略。

---

## 一、项目开发目标（整体与本次子目标）

### 1. 整体项目目标（shoplazza_dashboard / line chart）

在 **Shoplazza 多店铺**场景下，建设可运维的数据与运营能力，典型包括：

- **数据同步与统计**：根目录脚本（如 `data_sync`、`fill_*`）按北京时间拉单、入库与聚合；与店匠 API、限流与任务调度协同。
- **可视化与运营**：Streamlit/历史看板与 **Vue + FastAPI** 新管理后台并存或演进；看板、负责人汇总、映射与审计、**店铺运营/员工归因**、权限等。
- **可观测与排障**：日志、数据库迁移、配置（根目录 `config.py` / `backend/config_new.py`、`.env`）分层清晰，便于单机与后续集中式日志（如 ELK/Loki）扩展。

### 2. 本次「日志体系改造」子目标

| 目标 | 说明 |
|------|------|
| 告别单文件无限增长 | 不再依赖仅追加 `logs/app.log` 的 `FileHandler` |
| 按日落盘 + 按大小轮转 | 本地午夜或单文件超阈任一即轮转；历史段 **gzip**；约 **30 天**保留 |
| ERROR 分流 | `error-YYYY-MM-DD.log` 仅 ERROR 及以上 |
| 兼容现有代码 | 业务继续用 **`logging.getLogger(__name__)`**，经 **InterceptHandler** 进入 loguru |
| 根脚本与 API 一致 | **同一套** `setup_logging` 行为；后端不强行依赖根目录 `config.py`，由 **`config_new.LOG_CONFIG` 注入** |

## 二、开发过程概要

### 阶段 A（根目录与脚本）

1. 根目录 **`requirements.txt`** 增加 **`loguru>=0.7.0`**。  
2. **`config.py`** 扩展 **`LOG_CONFIG`**（`TypedDict`、`rotation_midnight`、`retention`、`enable_json_log`、`error_diagnose` 等），**`config_example.py`** 同步。  
3. 新增 **`lib/log_config.py`**：幂等初始化、`InterceptHandler`、combined/error 文件 sink、控制台彩色 format、第三方库降噪、可选 JSON sink。  
4. **`utils.setup_logging`** 改为调用 **`lib.log_config.setup_logging`**，并**传入**根目录 **`LOG_CONFIG`**，根脚本 **import 路径基本不变**。  
5. 验收：`fill_today_data.py` 等真实脚本跑通；combined 中为真实 `.py` 路径与行号；同进程连续两次 `setup_logging` 不重复落盘。

### 阶段 B（FastAPI / uvicorn）

1. **`backend/requirements.txt`** 增加 **`loguru>=0.7.0`**（与根目录下限一致）。  
2. **`lib/log_config.setup_logging`** 增加第三参 **`log_config`**（可选）；未传时仍读根目录 **`config.LOG_CONFIG`**，保证根脚本行为不变。  
3. **`backend/config_new.py`** 中 **`LOG_CONFIG`** 与根目录键对齐；**`log_dir` / `log_file`** 使用 **`_PROJECT_ROOT`**，日志统一落在**项目根 `logs/`**。  
4. **`backend/app/main.py`**：在文件前部将 **项目根**与 **`backend`** 加入 **`sys.path`**（仅为解析 **`lib`** 与 **`config_new`**，不 import 根目录 `config`）；调用 **`setup_logging(..., LOG_CONFIG)`**；去掉 **`logging.basicConfig`**；对 **`uvicorn` / `uvicorn.error` / `uvicorn.access`** 清空 handler 并 **`propagate=True`**，减轻与根拦截器的重复输出。  
5. 验收：访问 API 后 **`logs/combined-*.log`** 出现 **`app.main`** 与业务模块日志；幂等探测通过。

## 三、关键实现说明（便于交接）

| 项 | 说明 |
|----|------|
| 轮转 OR 条件 | loguru 0.7.x 对 `rotation=[...]` 列表解析不一致处，使用 **`_rotation_or_midnight_and_size`** 手工组合「午夜 OR 大小」。 |
| 行号与文件 | InterceptHandler 使用标准库 **`LogRecord.pathname` / `lineno`** 写入 **`extra`**，避免仅依赖 `opt(depth)` 在 `-c` 等场景出现 `<unknown>:0`。 |
| 后端配置解耦 | 后端 **`LOG_CONFIG` 全部由 `config_new` 传入**，不依赖根目录 **`config.py`**，降低目录结构变更时的耦合。 |
| 控制台「三行一组」 | 单次 HTTP 常见为：**中间件请求** + **中间件响应** + **`uvicorn.access`**，属不同来源，不是同一条日志重复打印。 |

## 四、验收与已知问题

### 已通过项（摘要）

- 项目根 **`logs/combined-当日.log`**、**`error-当日.log`** 正常写入；业务与 API 路径、行号可读。  
- 根目录与 **`backend/requirements.txt`** 中 **loguru** 版本下限一致。  
- 同进程 **两次 `setup_logging`** 不产生重复行。  

### Windows 下 pip 读 requirements 报错（UnicodeDecodeError / gbk）

- **现象**：`backend/requirements.txt` 中含 UTF-8 中文注释时，部分环境下 pip 按 GBK 解码失败。  
- **处理**：安装前执行 **`$env:PYTHONUTF8="1"`**（PowerShell），或将相关注释改为英文。
