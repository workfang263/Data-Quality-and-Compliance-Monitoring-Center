## 一、本窗口项目开发目标

本窗口目标聚焦在 C.2 落地后的“可用性和工程化收口”，具体是四件事：

1. 修复前端 TypeScript 构建错误，恢复 `npm run build` 可通过状态；
2. 修复配置中心三卡片 UI 显示不完整问题（表格列被挤压）；
3. 让配置中心“候选店铺/候选广告账户”口径与主系统“映射编辑页”一致；
4. 精简“新增运营”表单，只保留 `employee_slug`、`utm_keyword`、`campaign_keyword`。

## 三、本窗口遇到的问题与解决方法

### 问题 1：前端 TS 严格模式错误较多，构建失败
- **表现**：`vue-tsc -b` 报多处 TS6133 / TS2322 / TS18048 / TS7016 等错误。
- **根因**：历史组件缺少严格类型约束，且 `element-plus` locale 模块无声明文件。
- **解决**：逐文件修正类型签名、空值兜底、未使用变量；新增 `shims.d.ts` 模块声明。
- **结果**：构建恢复可通过。

### 问题 2：配置中心三卡片并排导致列被挤压，显示不完整
- **表现**：截图中的状态/操作列布局拥挤，视觉上“缺块”或截断。
- **根因**：三列布局下每个表格横向空间不足。
- **解决**：改为纵向平铺卡片，保留表格结构但给足宽度。
- **结果**：常见分辨率下表格完整展示。

### 问题 3：候选数量与映射编辑页不一致
- **表现**：广告账户只能选约 30+，店铺也少于映射编辑页。
- **根因**：旧接口额外做了业务过滤（如 `is_active=1`、排除已占用），与“映射编辑页全量口径”不一致。
- **解决**：候选接口改为“同源全量 + 绑定状态标记”，前端负责禁选已绑定项。
- **结果**：候选集合与映射编辑页对齐，同时不破坏“账户唯一绑定”的业务约束。

### 问题 4：新增运营只需三项，但后端原模型强制 `display_name`
- **表现**：前端即使精简表单，也会被后端校验拦截。
- **根因**：后端创建模型与新交互要求不一致。
- **解决**：后端把 `display_name` 改为可选，缺省自动等于 slug。
- **结果**：前端三字段表单可直接创建成功。
