# 实时聚合方案设计（最终版）

## 📋 当前情况分析

### 现有脚本
1. **`data_sync.py`** - 每5分钟执行一次，收集店铺数据（销售额、订单数、访客数）
2. **`fb_spend_sync.py`** - 需要创建5分钟定时任务版本，收集广告花费数据

### 执行方式
- 两个脚本都会每5分钟执行一次（Windows 定时任务）
- 两个脚本可能**同时运行**，也可能**不同时完成**
- 需要确保数据收集完成后**实时聚合**

### 关键问题
- 如果店铺数据先完成，广告花费还没完成 → 聚合时广告花费数据不完整
- 如果广告花费先完成，店铺数据还没完成 → 聚合时店铺数据不完整
- 如何确保聚合时数据完整？

---

## 💡 方案对比

### 方案1：每个脚本完成后都聚合（推荐⭐）

**实现方式**：
- `data_sync.py` 完成后，聚合今天的数据
- `fb_spend_sync.py` 完成后，也聚合今天的数据
- 使用 `REPLACE INTO`，重复聚合不会出错（幂等操作）

**代码示例**：
```python
# 在 data_sync.py 的最后
if summary_write_success:
    # ... 更新同步状态 ...
    
    # 聚合今天的数据
    try:
        from aggregate_owner_daily import aggregate_date
        today = datetime.date.today()
        aggregate_date(db.get_connection(), today, verbose=False)
        logger.info(f"聚合任务完成: {today}")
    except Exception as e:
        logger.error(f"聚合任务失败: {e}", exc_info=True)
```

```python
# 在 fb_spend_sync.py 的最后
upsert_spend(conn, rows)
print(f"[OK ] {since} 写入 {len(rows)} 条")

# 聚合今天的数据
try:
    from aggregate_owner_daily import aggregate_date
    today = datetime.date.today()
    aggregate_date(conn, today, verbose=False)
    print(f"[聚合] {today} 聚合完成")
except Exception as e:
    print(f"[ERR] {today} 聚合失败: {e}")

conn.close()
```

**优点**：
- ✅ **实现最简单**：只需要在两个脚本末尾添加几行代码
- ✅ **实时性好**：每个脚本完成后立即聚合
- ✅ **可靠性高**：
  - `REPLACE INTO` 是幂等操作，重复聚合不会出错
  - 即使两个脚本不同时完成，最终数据也会正确
  - 如果某个脚本失败，另一个脚本的聚合仍然会执行
- ✅ **容错性好**：聚合失败不影响主流程

**缺点**：
- ⚠️ 可能重复聚合（但 `REPLACE INTO` 是幂等的，不影响结果）
- ⚠️ 如果两个脚本都成功，会聚合两次（但性能影响可忽略，<1秒）

**适用场景**：**推荐使用**，简单可靠

### 方案2：使用数据库标记，确保两个脚本都完成后再聚合

**实现方式**：
- 创建标记表 `sync_completion_flags`，记录每个脚本的完成时间
- 每个脚本完成后，更新自己的标记
- 检查两个脚本都完成后，才执行聚合

**代码示例**：
```sql
CREATE TABLE sync_completion_flags (
    sync_type VARCHAR(50) PRIMARY KEY,
    last_completion_time DATETIME,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

```python
# 在 data_sync.py 的最后
if summary_write_success:
    # 更新完成标记
    db.update_sync_completion_flag('store_sync', datetime.now())
    
    # 检查两个脚本是否都完成
    if db.check_both_syncs_completed():
        # 聚合今天的数据
        aggregate_date(db.get_connection(), today, verbose=False)
```

**优点**：
- ✅ 确保数据完整后再聚合
- ✅ 避免重复聚合

**缺点**：
- ❌ **实现复杂**：需要创建标记表，需要检查逻辑
- ❌ **实时性差**：必须等两个脚本都完成，可能有延迟
- ❌ **容错性差**：如果某个脚本失败，聚合可能永远不执行
- ❌ **维护成本高**：需要管理标记表

**适用场景**：不推荐，过于复杂

### 方案3：统一聚合脚本（独立定时任务）

**实现方式**：
- 创建独立的聚合脚本 `aggregate_realtime.py`
- 每5分钟执行一次，聚合今天的数据
- 与数据收集脚本独立运行

**代码示例**：
```python
# aggregate_realtime.py
def main():
    today = datetime.date.today()
    conn = get_db_conn()
    try:
        aggregate_date(conn, today, verbose=False)
    finally:
        conn.close()
```

**Windows 定时任务**：
- 每5分钟执行一次 `python aggregate_realtime.py`

**优点**：
- ✅ 完全解耦，不影响数据收集脚本
- ✅ 简单直接

**缺点**：
- ⚠️ **实时性差**：最多延迟5分钟（如果刚聚合完，数据更新了，要等5分钟）
- ⚠️ **可能聚合不完整**：如果数据还在收集中，聚合时数据不完整
- ⚠️ **需要额外的定时任务**

**适用场景**：如果对实时性要求不高，可以使用

### 方案4：异步聚合（后台线程）

**实现方式**：
- 使用后台线程执行聚合
- 每个脚本完成后，启动后台线程聚合

**优点**：
- ✅ 不阻塞主流程
- ✅ 实时性好

**缺点**：
- ❌ **实现复杂**：需要管理线程池
- ❌ **可能重复聚合**：两个脚本可能同时启动聚合线程
- ❌ **调试困难**：异步执行，错误不容易追踪

**适用场景**：不推荐，过于复杂

## 🎯 最终推荐：方案1（每个脚本完成后都聚合）

### 为什么推荐方案1？

1. **实现最简单**：只需要在两个脚本末尾添加几行代码
2. **实时性最好**：每个脚本完成后立即聚合，延迟<1秒
3. **可靠性最高**：
   - `REPLACE INTO` 是幂等操作，重复聚合不会出错
   - 即使两个脚本不同时完成，最终数据也会正确
   - 如果某个脚本失败，另一个脚本的聚合仍然会执行
4. **容错性好**：聚合失败不影响主流程

### 关于重复聚合的说明

- **不会出错**：`REPLACE INTO` 是幂等操作，重复执行结果相同
- **性能影响可忽略**：聚合通常<1秒，重复一次影响很小
- **数据最终一致**：即使两个脚本不同时完成，最终数据也会正确
