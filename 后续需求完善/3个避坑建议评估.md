## ✅ 建议1：补全脚本的锁文件复用

### 建议内容

**问题描述：**
- `fill_today_data.py` 或 `fill_missing_data.py` 可能被手动运行
- 如果10:00的定时任务正在跑，你正好手动运行了补全脚本
- 由于它们访问的是同样的表且都需要做"查旧值+写新值"的操作，依然可能发生竞争

**优化建议：**
- 给这些补全脚本也加上同样的 `acquire_lock()` 逻辑
- 指向同一个 `data_sync.lock` 文件
- 让它们共享同一个锁文件，确保"手动补录"和"自动同步"永远不会在同一秒钟打架

### 评估：✅ **强烈推荐采纳**

**理由：**
- ✅ **问题真实存在**：补全脚本和自动同步脚本会访问相同的数据库表，存在竞争条件
- ✅ **解决方案简单**：只需要在补全脚本的开头添加相同的进程锁逻辑
- ✅ **成本低、收益高**：几行代码就能彻底解决潜在的竞争问题
- ✅ **符合最佳实践**：所有可能修改同一资源的脚本都应该使用相同的锁机制

**需要修改的文件：**
- `fill_today_data.py`
- `fill_missing_data.py`
- `fill_date_data.py`（如果也会修改实时同步相关的表）

**实现方式：**
```python
# 在 fill_today_data.py 的开头添加
import msvcrt
import sys
import os

def acquire_lock():
    """获取进程锁（与 data_sync.py 使用相同的锁文件）"""
    script_dir = os.path.dirname(os.path.abspath(__file__))
    lock_file_path = os.path.join(script_dir, "data_sync.lock")  # 使用相同的锁文件
    
    try:
        lock_file = open(lock_file_path, 'a')
        msvcrt.locking(lock_file.fileno(), msvcrt.LK_NBLCK, 1)
        return lock_file
    except IOError:
        print("⚠️  检测到另一个进程正在同步中（可能是 data_sync.py 或另一个补全脚本），当前进程自动退出。")
        sys.exit(0)
    except Exception as e:
        print(f"❌ 获取进程锁失败: {e}")
        sys.exit(1)

# 在主函数中
def fill_today_data():
    # 获取进程锁
    lock_handle = acquire_lock()
    
    try:
        # 主逻辑...
        ...
    finally:
        if lock_handle:
            lock_handle.close()
```

**注意事项：**
- ⚠️ **使用相同的锁文件路径**：必须使用 `data_sync.lock`，确保所有脚本共享同一个锁
- ⚠️ **锁文件路径一致性**：确保所有脚本都使用相同的路径计算方法（`os.path.dirname(os.path.abspath(__file__))`）

## ✅ 建议2：数据库 time_hour 索引的确认

### 建议内容

**问题描述：**
- 改成了覆盖式写入（`ON DUPLICATE KEY UPDATE`）
- 如果没有唯一索引，`ON DUPLICATE KEY UPDATE` 就不会生效
- 数据库会不停地插入新行，导致看到几十个相同小时的重复数据

**优化建议：**
- 确认 `shoplazza_store_hourly` 表必须有 `UNIQUE KEY`，包含 `(shop_domain, time_hour)` 这两个字段
- 确认 `shoplazza_overview_hourly` 表必须有 `UNIQUE KEY`，包含 `(time_hour)` 字段

### 评估：✅ **强烈推荐采纳（必须验证）**

**理由：**
- ✅ **问题极其严重**：没有唯一索引，`ON DUPLICATE KEY UPDATE` 完全不会生效
- ✅ **后果严重**：会导致数据重复插入，数据库中会有多个相同时间点的记录
- ✅ **必须验证**：这是一个关键的前置条件，必须在实施前验证

**验证方法：**

**方法1：查看表结构（SQL）**
```sql
-- 查看 shoplazza_store_hourly 表的索引
SHOW CREATE TABLE shoplazza_store_hourly;

-- 或者使用以下查询
SELECT 
    TABLE_NAME,
    INDEX_NAME,
    COLUMN_NAME,
    NON_UNIQUE
FROM 
    information_schema.STATISTICS
WHERE 
    TABLE_SCHEMA = DATABASE()
    AND TABLE_NAME IN ('shoplazza_store_hourly', 'shoplazza_overview_hourly')
    AND NON_UNIQUE = 0
ORDER BY 
    TABLE_NAME, INDEX_NAME, SEQ_IN_INDEX;
```

**方法2：检查是否有重复数据（验证）**
```sql
-- 检查 shoplazza_overview_hourly 表中是否有重复的 time_hour
SELECT 
    time_hour, 
    COUNT(*) as count
FROM shoplazza_overview_hourly
GROUP BY time_hour
HAVING COUNT(*) > 1
LIMIT 10;

-- 检查 shoplazza_store_hourly 表中是否有重复的 (shop_domain, time_hour)
SELECT 
    shop_domain,
    time_hour, 
    COUNT(*) as count
FROM shoplazza_store_hourly
GROUP BY shop_domain, time_hour
HAVING COUNT(*) > 1
LIMIT 10;
```

**如果发现没有唯一索引，需要添加：**

```sql
-- 为 shoplazza_overview_hourly 表添加唯一索引（如果不存在）
ALTER TABLE shoplazza_overview_hourly
ADD UNIQUE KEY uk_time_hour (time_hour);

-- 为 shoplazza_store_hourly 表添加唯一索引（如果不存在）
ALTER TABLE shoplazza_store_hourly
ADD UNIQUE KEY uk_shop_time (shop_domain, time_hour);
```

**注意事项：**
- ⚠️ **如果表中已有重复数据**：添加唯一索引会失败，需要先清理重复数据
- ⚠️ **清理重复数据的方法**：
  ```sql
  -- 对于 shoplazza_overview_hourly，保留最新的记录
  DELETE t1 FROM shoplazza_overview_hourly t1
  INNER JOIN shoplazza_overview_hourly t2
  WHERE t1.time_hour = t2.time_hour
    AND t1.id < t2.id;
  
  -- 对于 shoplazza_store_hourly，保留最新的记录
  DELETE t1 FROM shoplazza_store_hourly t1
  INNER JOIN shoplazza_store_hourly t2
  WHERE t1.shop_domain = t2.shop_domain
    AND t1.time_hour = t2.time_hour
    AND t1.id < t2.id;
  ```

## ✅ 建议3：跨天补录的极端情况

### 建议内容

**问题描述：**
- 在 00:10:00 执行时，逻辑会收集昨天 23:50:00 - 23:59:59 的数据
- 如果 `sync_yesterday_final_data()` 在 00:00:00 执行并更新了状态，00:10:00 的脚本可能会认为 23:50 的段已经同步过了
- 导致昨天的最后10分钟段被跳过

**优化建议：**
- 确保 `sync_yesterday_final_data` 处理完后，将 `last_sync_end_time` 准确地标记为昨天的 23:59:59

### 评估：✅ **强烈推荐采纳（需要修复）**

**当前代码分析：**

查看 `data_sync.py` 的 `sync_yesterday_final_data()` 函数（第1346行）：
```python
# 4. 更新同步状态：标记昨天数据已完成
db.update_sync_status('five_minute_realtime', yesterday_end, yesterday_start.date(), total_visitors)
```

其中 `yesterday_end = yesterday_start.replace(hour=23, minute=59, second=59, microsecond=999999)`

**评估：**
- ✅ **当前逻辑基本正确**：`yesterday_end` 已经是 `23:59:59.999999`，这是正确的
- ⚠️ **但需要考虑10分钟段的边界**：如果改为10分钟间隔，应该标记为 `23:59:59`（而不是 `23:59:59.999999`）？

**关键问题：**
- 当前代码使用的是 `23:59:59.999999`，这对于5分钟段是正确的（23:55:00 - 23:59:59）
- 如果改为10分钟段，昨天最后一个段应该是 23:50:00 - 23:59:59，标记为 `23:59:59` 是正确的

**需要修复的点：**

1. **确保标记的是最后一个10分钟段的结束时间**
   - 如果是10分钟段，应该标记为 `23:59:59`（而不是 `23:59:59.999999`）
   - 但 `23:59:59.999999` 也可以，因为 00:10:00 的脚本会比较 `23:59:59.999999 > 23:59:59` 是 false，所以会正确跳过

2. **更关键的问题：sync_type 的名称**
   - 当前代码使用的是 `'five_minute_realtime'`
   - 如果改为10分钟，应该使用 `'ten_minute_realtime'`
   - 或者：`sync_yesterday_final_data()` 也应该使用新的 `sync_type`

**建议修复：**

```python
def sync_yesterday_final_data():
    """
    在00:00:00执行时，收集昨天的最终数据
    
    ⚠️ 注意：如果是10分钟间隔，应该收集 23:50:00-23:59:59 的数据
    """
    db = Database()
    now = beijing_time()
    yesterday_start = (now - timedelta(days=1)).replace(hour=0, minute=0, second=0, microsecond=0)
    yesterday_end = yesterday_start.replace(hour=23, minute=59, second=59, microsecond=999999)
    
    # ... 收集数据的逻辑 ...
    
    # ⭐ 修复：确保标记为最后一个10分钟段的结束时间（23:59:59）
    # 对于10分钟段，最后一个段是 23:50:00-23:59:59
    last_segment_end = yesterday_start.replace(hour=23, minute=59, second=59, microsecond=0)
    
    # ⭐ 修复：使用新的 sync_type（如果改为10分钟间隔）
    sync_type = 'ten_minute_realtime'  # 或者继续使用 'five_minute_realtime'，但建议使用新的
    db.update_sync_status(sync_type, last_segment_end, yesterday_start.date(), total_visitors)
```

**或者更简单的方案（推荐）：**

```python
# 确保标记为最后一个10分钟段的结束时间（精确到秒，不需要微秒）
last_segment_end = yesterday_start.replace(hour=23, minute=59, second=59, microsecond=0)

# 更新同步状态
sync_type = 'ten_minute_realtime'  # 使用新的 sync_type
db.update_sync_status(sync_type, last_segment_end, yesterday_start.date(), total_visitors)
```

**验证方法：**

在部署后，检查 `sync_status` 表：
```sql
SELECT * FROM sync_status WHERE sync_type = 'ten_minute_realtime';
```

确认 `last_sync_end_time` 是否为 `23:59:59`（昨天的最后一个10分钟段的结束时间）
