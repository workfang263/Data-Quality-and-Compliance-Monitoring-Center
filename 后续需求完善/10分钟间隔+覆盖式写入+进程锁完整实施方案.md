# 10分钟间隔+覆盖式写入+进程锁完整实施方案

## 📋 方案概述

本方案结合进程锁、覆盖式写入和任务自愈能力，彻底解决数据翻倍问题，提升系统稳定性和健壮性。

---

## 🎯 核心优化点

| 优化项 | 当前状态 | 优化后 | 解决什么问题 |
|--------|---------|--------|-------------|
| **执行频率** | 每5分钟 | 每10分钟 | 降低50%系统负载，减少进程重叠 |
| **写入模式** | 累加式 | 覆盖式 | 即使进程重叠也不会翻倍 |
| **进程保护** | 无 | msvcrt进程锁 | 物理层面防止多进程同时运行 |
| **补全脚本锁** | 无 | 共享进程锁 | 防止补全脚本与自动同步脚本竞争 |
| **数据库索引** | 需验证 | 确保唯一索引存在 | 确保覆盖式写入生效 |
| **跨天补录** | 需修复 | 正确的sync_type和结束时间 | 防止数据缺失 |
| **任务自愈** | 部分实现 | 连续补录缺失段 | 自动恢复，无需手动干预 |
| **性能优化** | 基础实现 | Owner缓存等 | 减少数据库查询，提升性能 |

### 1. 进程锁（Process Lock）实现

#### 1.1 实现位置

在 `data_sync.py` 文件的最顶部（所有导入语句之后，main函数之前）添加。

#### 1.2 代码实现

```python
import msvcrt
import sys
import os

def acquire_lock():
    """
    获取进程锁（防止多个进程同时运行）
    
    Returns:
        file: 锁文件句柄（必须返回并保持打开状态，否则锁会失效）
    
    如果获取锁失败（另一个进程正在运行），直接退出程序
    """
    # 在脚本目录下创建锁文件
    script_dir = os.path.dirname(os.path.abspath(__file__))
    lock_file_path = os.path.join(script_dir, "data_sync.lock")
    
    try:
        # 以追加模式打开（不破坏文件）
        lock_file = open(lock_file_path, 'a')
        
        # LK_NBLCK 是非阻塞锁：如果锁不住，立即抛出IOError而不是等待
        msvcrt.locking(lock_file.fileno(), msvcrt.LK_NBLCK, 1)
        
        # ⚠️ 重要：必须返回文件句柄，否则函数结束文件关闭，锁就失效了
        return lock_file
        
    except IOError:
        # 另一个进程正在运行，当前进程退出
        print("⚠️  检测到另一个进程正在同步中，当前进程自动退出。")
        logger.info("进程锁获取失败，另一个进程正在运行，退出")
        sys.exit(0)
    except Exception as e:
        # 其他异常（例如文件权限问题）
        logger.error(f"获取进程锁失败: {e}")
        print(f"❌ 获取进程锁失败: {e}")
        sys.exit(1)
```

#### 1.3 使用方式

在 `main()` 函数中，在执行主逻辑前获取锁：

```python
def main():
    # 获取进程锁（必须在主逻辑开始前）
    lock_handle = acquire_lock()
    
    try:
        # 主逻辑...
        if '--realtime' in sys.argv:
            sync_realtime_data_ten_minutes()
        # ...其他逻辑
        
    finally:
        # 确保锁被释放（虽然程序结束会自动释放，但显式关闭更安全）
        if lock_handle:
            lock_handle.close()
            logger.info("进程锁已释放")
```

#### 1.4 注意事项

- ✅ **文件句柄必须保持打开**：锁的生命周期与文件句柄绑定，文件关闭锁就失效
- ✅ **使用 try...finally**：确保即使发生异常，锁也能被释放
- ✅ **锁文件位置**：放在脚本目录下（`data_sync.lock`），便于管理
- ⚠️ **异常处理**：如果获取锁失败，直接退出，不要继续执行

### 3. 覆盖式写入（覆盖式写入）实现

#### 3.1 核心逻辑：读取-计算-覆盖

**当前逻辑（累加式）：**
```python
# 直接累加写入
db.insert_or_update_hourly_data_incremental(
    time_hour, total_sales, total_orders, total_visitors
)
```

**优化后逻辑（覆盖式）：**
```python
# 1. 查询数据库：获取当前小时已有的总销售额和订单数
existing_data = db.get_hourly_data_by_time(time_hour)

# 2. API采集：并发收集35个店铺这10分钟的增量（已过滤COD等）
new_sales = 本次API采集的销售额
new_orders = 本次API采集的订单数
new_visitors = 本次API采集的访客数

# 3. 合并数据：新总值 = 数据库已有值 + API增量值
if existing_data:
    total_sales = existing_data['total_gmv'] + new_sales
    total_orders = existing_data['total_orders'] + new_orders
    total_visitors = new_visitors  # 访客数直接覆盖（因为API返回的是当天累计去重值）
else:
    # 首次写入（该小时还没有数据）
    total_sales = new_sales
    total_orders = new_orders
    total_visitors = new_visitors

# 4. 覆盖写入：使用 ON DUPLICATE KEY UPDATE gmv = VALUES(gmv)（不带加号）
avg_order_value = total_sales / total_orders if total_orders > 0 else 0.0
db.insert_or_update_hourly_data(
    time_hour, total_sales, total_orders, total_visitors, avg_order_value
)
```

#### 3.2 为什么使用覆盖式写入？

**关键优势：**
- ✅ **幂等性**：即使重复执行，结果也是一样的
- ✅ **防止翻倍**：即使进程锁失效，覆盖写入也不会导致数据翻倍
- ✅ **易于调试**：每次写入都是完整的小时数据，易于验证
- ✅ **兼容补全脚本**：即使手动运行补齐脚本，也不会导致数据翻倍

**对比：**
- **累加式**：`total_gmv = total_gmv + VALUES(total_gmv)` → 重复执行会累加
- **覆盖式**：`total_gmv = VALUES(total_gmv)` → 重复执行结果相同

### 4. 任务自愈（连续补录缺失段）

#### 4.1 当前逻辑的问题

**当前逻辑：**
- 如果时间差超过30分钟，只收集最近1小时的数据
- 问题：如果服务器断网30分钟，恢复后只补最近1小时，前面的数据缺失

#### 4.2 优化后的逻辑

**优化后逻辑：连续补录所有缺失的10分钟段**

```python
def sync_realtime_data_ten_minutes():
    """十分钟实时数据同步主函数"""
    db = Database()
    now = beijing_time()
    today_start = now.replace(hour=0, minute=0, second=0, microsecond=0)
    
    # ... 跨天处理 ...
    
    # 1. 计算当前应该同步的10分钟段
    current_minute = now.minute
    current_segment_start_minute = (current_minute // 10) * 10
    current_segment_start = now.replace(minute=current_segment_start_minute, second=0, microsecond=0)
    current_segment_end = current_segment_start + timedelta(minutes=10) - timedelta(microseconds=1)
    
    # 判断当前段是否已完成
    if now >= current_segment_end:
        recent_10min_start = current_segment_start
        recent_10min_end = current_segment_end
    else:
        # 当前段未完成，计算最近完成的段
        recent_10min_start_minute = current_segment_start_minute - 10
        if recent_10min_start_minute < 0:
            recent_10min_start_minute += 60
            recent_10min_start_hour = now.hour - 1
        else:
            recent_10min_start_hour = now.hour
        
        recent_10min_start = now.replace(
            hour=recent_10min_start_hour,
            minute=recent_10min_start_minute,
            second=0,
            microsecond=0
        )
        recent_10min_end = recent_10min_start + timedelta(minutes=10) - timedelta(microseconds=1)
    
    # 2. 获取最后同步时间
    sync_status = db.get_sync_status('ten_minute_realtime')  # 注意：sync_type需要更新
    if sync_status:
        last_sync_end_time = sync_status['last_sync_end_time']
        
        # 如果最后同步时间不是今天的，从今天00:00:00开始
        if last_sync_end_time.date() != today_start.date():
            next_segment_start = today_start
        else:
            # 计算下一个应该同步的段的开始时间
            next_segment_start = last_sync_end_time + timedelta(seconds=1)
            next_segment_start = next_segment_start.replace(second=0, microsecond=0)
            # 对齐到10分钟边界
            next_segment_minute = (next_segment_start.minute // 10) * 10
            next_segment_start = next_segment_start.replace(minute=next_segment_minute, second=0, microsecond=0)
    else:
        # 首次运行：从最近完成的段开始
        next_segment_start = recent_10min_start
    
    # 3. 连续补录所有缺失的段
    segments_to_process = []
    segment_start = next_segment_start
    
    while segment_start <= recent_10min_end:
        segment_end = segment_start + timedelta(minutes=10) - timedelta(microseconds=1)
        # 只处理已完成的段
        if segment_end <= now:
            segments_to_process.append((segment_start, segment_end))
        segment_start = segment_end + timedelta(seconds=1)
        segment_start = segment_start.replace(second=0, microsecond=0)
        segment_minute = (segment_start.minute // 10) * 10
        segment_start = segment_start.replace(minute=segment_minute, second=0, microsecond=0)
    
    if not segments_to_process:
        logger.info("没有新数据需要收集")
        return
    
    logger.info(f"发现 {len(segments_to_process)} 个缺失的10分钟段需要补录")
    
    # 4. 依次处理每个缺失的段
    for segment_start, segment_end in segments_to_process:
        logger.info(f"处理10分钟段: {segment_start} - {segment_end}")
        success = process_ten_minute_segment(segment_start, segment_end, db)
        
        if success:
            # 更新同步状态
            sync_date = segment_start.date()
            db.update_sync_status('ten_minute_realtime', segment_end, sync_date, 0)
            logger.info(f"10分钟段处理完成: {segment_start} - {segment_end}")
        else:
            logger.error(f"10分钟段处理失败: {segment_start} - {segment_end}")
            # 如果某个段处理失败，停止后续段的处理（避免数据不连续）
            break
```

#### 4.3 提取处理单个段的逻辑

将处理单个10分钟段的逻辑提取为独立函数：

```python
def process_ten_minute_segment(start_time: datetime, end_time: datetime, db: Database) -> bool:
    """
    处理单个10分钟段的数据收集和写入
    
    Args:
        start_time: 段开始时间
        end_time: 段结束时间
        db: 数据库实例
    
    Returns:
        是否成功
    """
    try:
        # 1. 获取所有启用的店铺
        stores = db.get_active_stores()
        if not stores:
            logger.warning("没有启用的店铺，跳过同步")
            return False
        
        # 2. 预先加载Owner信息（性能优化）
        owner_map = {}
        for store in stores:
            shop_domain = store['shop_domain']
            owner = db.get_store_display_name(shop_domain)  # 或者从store对象获取
            owner_map[shop_domain] = owner
        
        # 3. 并行收集所有店铺的数据（10个线程）
        all_store_data = {}
        with concurrent.futures.ThreadPoolExecutor(max_workers=10) as executor:
            future_to_store = {
                executor.submit(sync_store_data_for_ten_minutes, 
                              store['shop_domain'], store['access_token'],
                              start_time, end_time): store
                for store in stores
            }
            
            for future in concurrent.futures.as_completed(future_to_store):
                store = future_to_store[future]
                try:
                    result = future.result()
                    all_store_data[store['shop_domain']] = result
                except Exception as e:
                    logger.error(f"店铺 {store['shop_domain']} 数据收集异常: {e}")
                    all_store_data[store['shop_domain']] = {
                        'success': False,
                        'sales': 0.0,
                        'orders': 0,
                        'visitors': 0,
                        'error': str(e)
                    }
        
        # 4. 汇总所有店铺的数据
        total_sales = 0.0
        total_orders = 0
        total_visitors = 0
        
        # 确定数据属于哪个小时
        time_hour = start_time.replace(minute=0, second=0, microsecond=0)
        
        # 5. 写入单店铺明细表（覆盖式）
        for shop_domain, data in all_store_data.items():
            if data['success']:
                # 查询已有数据
                existing_store_data = db.get_store_hourly_data(
                    shop_domain, time_hour, time_hour + timedelta(hours=1) - timedelta(seconds=1)
                )
                
                if existing_store_data and len(existing_store_data) > 0:
                    existing = existing_store_data[0]
                    store_total_sales = existing['total_gmv'] + data['sales']
                    store_total_orders = existing['total_orders'] + data['orders']
                    store_total_visitors = data['visitors']
                else:
                    store_total_sales = data['sales']
                    store_total_orders = data['orders']
                    store_total_visitors = data['visitors']
                
                # 覆盖写入
                owner = owner_map.get(shop_domain, '')
                db.insert_or_update_store_hourly(
                    shop_domain, time_hour, store_total_sales, store_total_orders,
                    store_total_visitors
                )
                
                # 累加到总数据
                total_sales += data['sales']
                total_orders += data['orders']
                total_visitors += data['visitors']
        
        # 6. 写入汇总表（覆盖式）
        existing_data = db.get_hourly_data_by_time(time_hour)
        if existing_data:
            total_sales = existing_data['total_gmv'] + total_sales
            total_orders = existing_data['total_orders'] + total_orders
            total_visitors = total_visitors  # 覆盖
        # else: 首次写入，直接使用total_sales和total_orders
        
        avg_order_value = total_sales / total_orders if total_orders > 0 else 0.0
        success = db.insert_or_update_hourly_data(
            time_hour, total_sales, total_orders, total_visitors, avg_order_value
        )
        
        if success:
            logger.info(f"成功更新小时数据: {time_hour}, "
                       f"销售额=${total_sales:.2f}, 订单数={total_orders}, 访客数={total_visitors}")
            return True
        else:
            logger.error(f"更新小时数据失败: {time_hour}")
            return False
            
    except Exception as e:
        logger.error(f"处理10分钟段失败 {start_time} - {end_time}: {e}")
        return False
```

## ⚠️ 注意事项

### 1. 数据库索引验证（最关键！）

- **⚠️ 必须验证**：在实施前必须验证表结构是否有唯一索引
- **如果缺少索引**：覆盖式写入方案完全失效，会导致数据重复插入
- **如果表中有重复数据**：必须先清理重复数据，才能添加唯一索引
- **验证方法**：使用步骤2中的SQL查询验证

### 2. 进程锁文件

- **位置**：`data_sync.lock` 文件会在脚本目录下创建
- **共享**：所有脚本（`data_sync.py`、`fill_today_data.py`、`fill_missing_data.py` 等）使用同一个锁文件
- **清理**：正常情况下，脚本结束会自动释放锁
- **异常情况**：如果脚本异常退出，锁文件可能残留，需要手动删除（或设置自动清理机制）
- **锁文件路径一致性**：确保所有脚本使用相同的锁文件路径计算方法

### 2. sync_status表迁移

- **选项1**：使用新的 `sync_type` 名称（`'ten_minute_realtime'`）
  - 优点：更清晰，便于区分新旧逻辑
  - 缺点：需要更新代码中所有引用 `sync_type` 的地方

- **选项2**：继续使用旧的 `sync_type` 名称（`'five_minute_realtime'`）
  - 优点：无需修改数据库，向后兼容
  - 缺点：名称不准确

**建议**：使用新的 `sync_type` 名称，更清晰。

### 3. 跨天补录逻辑修复

- **确保 `sync_yesterday_final_data()` 使用正确的 `sync_type`**：`'ten_minute_realtime'`
- **确保标记的结束时间正确**：最后一个10分钟段的结束时间（`23:59:59`）
- **验证方法**：在部署后检查 `sync_status` 表，确认 `last_sync_end_time` 是否正确

### 4. 补全脚本的进程锁

- **所有补全脚本必须使用相同的锁文件**：`data_sync.lock`
- **确保锁文件路径一致性**：所有脚本使用相同的路径计算方法
- **测试验证**：同时运行自动同步和补全脚本，验证锁机制是否生效

### 5. 数据迁移（如果需要）

- **如果使用新的 `sync_type`**：旧数据不受影响，新数据使用新的 `sync_type`
- **如果继续使用旧的 `sync_type`**：无需数据迁移
