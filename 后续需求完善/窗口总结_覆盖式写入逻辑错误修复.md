### 步骤1：查看日志

- 检查17:40之后的日志，看是否有错误
- 发现日志正常，但数据不匹配

### 步骤2：分析代码逻辑

- 检查 `process_ten_minute_segment()` 函数
- 发现覆盖式写入逻辑错误：
  ```python
  existing_data = db.get_hourly_data_by_time(time_hour)
  if existing_data:
      total_sales = existing_gmv + total_sales  # ❌ 错误！
      total_orders = existing_orders + total_orders  # ❌ 错误！
  ```

### 步骤3：定位根本原因

**问题原因：**
- 同一个小时有多个10分钟段
- 每个段处理时都会读取数据库已有值并累加
- 如果同一个段被重复处理，会导致数据翻倍

## ✅ 修复方案

### 方案选择：增量累加模式

**优点：**
- ✅ 数据库层面原子操作，避免竞争条件
- ✅ 即使同一个段被处理多次，也不会翻倍
- ✅ 逻辑简单，易于维护

### 修复内容

1. **汇总表写入逻辑修复**
   - 改为使用 `insert_or_update_hourly_data_incremental()`
   - 在数据库层面累加，避免重复累加问题

2. **单店铺明细表写入逻辑修复**
   - 改为使用 `insert_or_update_store_hourly_incremental()`
   - 同样在数据库层面累加

## 🎓 关键知识点

### 1. 覆盖式写入 vs 增量累加

**覆盖式写入的问题：**
- 需要先查询已有数据，然后合并，最后覆盖
- 如果同一个段被处理多次，会导致重复累加
- 逻辑复杂，容易出错

**增量累加的优势：**
- 数据库层面原子操作，避免竞争条件
- 即使重复执行也不会翻倍
- 逻辑简单，易于维护

### 2. 测试开发工程师的测试方法

**测试金字塔：**
- 单元测试 > 集成测试 > 端到端测试

**测试原则：**
- 测试隔离：每个测试应该独立
- 测试可重复性：每次运行结果应该一致
- 测试覆盖：应该测试正常流程、边界情况、异常情况、并发情况

### 3. Mock 数据的使用

**为什么使用 Mock 数据？**
- 隔离测试：不依赖真实的 API
- 可控性：可以精确控制测试数据
- 速度：测试运行更快

**如何 Mock 数据？**
- 使用 `unittest.mock.patch` 装饰器
- 使用固定的随机种子确保可重复性
- 预先计算数据，确保期望值和实际值一致
