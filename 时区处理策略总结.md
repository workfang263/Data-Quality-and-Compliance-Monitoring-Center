# 时区处理策略总结

## 各平台API返回的时间格式分析

### 1. Shoplazza API

**返回格式：**
- `placed_at`: `2025-12-17T03:47:12Z`（UTC时间，Z格式）

**当前代码处理：**
- 识别 `Z` 为UTC时间
- 转换为北京时间（UTC+8）

**关键问题：**
- API返回的UTC时间是否已经考虑了店铺时区？
- 如果店铺时区是UTC-8，订单在UTC-8的 `00:00:00` 支付
- API应该返回UTC的 `08:00:00Z`（UTC-8的00:00 = UTC的08:00）
- 代码转换为北京时间 `16:00:00`（正确）

**结论：**
- ✅ **如果API返回的UTC时间已经考虑了店铺时区，可以不改代码**
- ⚠️ **但建议还是加上时区配置，更清晰明确**
- ❌ **如果API返回的UTC时间没有考虑店铺时区，必须修改代码**

---

### 2. Facebook API

**返回格式：**
- `date_start`: `2025-12-16`（账户时区的日期）
- `hourly_stats_aggregated_by_advertiser_time_zone`: `00:00:00 - 00:59:59`（账户时区的小时）

**当前代码处理：**
- 直接将 `date_start` 和 `hour_start` 拼接
- **没有时区转换**
- 直接当作北京时间存储

**关键问题：**
- 如果账户时区是UTC-8，返回的 `00:00:00` 是UTC-8的00:00
- 代码当作UTC+8存储，会导致时间错位16小时

**结论：**
- ❌ **必须修改代码，根据账户时区配置进行转换**
- ❌ **不能自动转换，因为API返回的是账户时区的时间，不是UTC时间**

---

### 3. TikTok API

- 如果返回的是账户时区的时间，需要修改代码
