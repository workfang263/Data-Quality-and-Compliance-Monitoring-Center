# 综合优化方案：10分钟间隔 + 覆盖式写入

## 📋 方案概述

结合两种优化方案，从根本上解决数据翻倍问题并提升系统稳定性：

1. **改为10分钟间隔执行**（从5分钟改为10分钟）
2. **改为覆盖式写入**（从累加式改为覆盖式）

---

## 🎯 优化目标

### 当前问题
- ❌ **数据翻倍**：竞态条件导致数据重复累加
- ❌ **系统负载高**：每5分钟执行一次，频率较高
- ❌ **竞态条件风险**：执行频率高，进程重叠概率大

### 优化后效果
- ✅ **彻底解决数据翻倍**：覆盖式写入 + 降低执行频率双重保障
- ✅ **降低系统负载**：执行频率减少50%（从12次/小时 → 6次/小时）
- ✅ **提升系统稳定性**：降低竞态条件发生的概率
- ✅ **数据延迟可接受**：10分钟的数据延迟对大多数业务场景可接受

---

## 📊 对比分析

| 项目 | 当前方案（5分钟+累加式） | 优化方案（10分钟+覆盖式） |
|------|------------------------|------------------------|
| **执行频率** | 每5分钟（12次/小时） | 每10分钟（6次/小时） |
| **数据延迟** | 约5-10分钟 | 约10-20分钟 |
| **写入模式** | 累加式（`total_gmv = total_gmv + VALUES(total_gmv)`） | 覆盖式（`total_gmv = VALUES(total_gmv)`） |
| **竞态条件风险** | 较高（执行频率高） | 很低（执行频率低 + 覆盖写入） |
| **数据翻倍风险** | 存在（累加写入 + 进程重叠） | 几乎不存在（覆盖写入） |
| **系统负载** | 较高 | 较低（减少50%） |
| **数据库查询次数** | 每次写入前0次（累加模式） | 每次写入前1次（查询已有数据） |
| **数据准确性** | 易受竞态条件影响 | 高（每次写入完整小时数据） |
| **可维护性** | 中等 | 高（数据更清晰，易于调试） |

## 🔧 技术实现

### 1. 改为10分钟间隔

#### 需要修改的文件

**文件1：`data_sync.py`**

主要修改点：
- 函数名：`sync_realtime_data_five_minutes()` → `sync_realtime_data_ten_minutes()`
- 时间段计算：从 `// 5 * 5` 改为 `// 10 * 10`
- 时间段长度：从 `timedelta(minutes=5)` 改为 `timedelta(minutes=10)`
- 跨天处理：从00:05改为00:10
- 函数注释和日志信息更新

**关键代码修改：**

```python
# 原代码（5分钟）：
def sync_realtime_data_five_minutes():
    current_segment_start_minute = (current_minute // 5) * 5
    current_segment_end = current_segment_start + timedelta(minutes=5) - timedelta(microseconds=1)
    recent_5min_start_minute = current_segment_start_minute - 5

# 改为（10分钟）：
def sync_realtime_data_ten_minutes():
    current_segment_start_minute = (current_minute // 10) * 10
    current_segment_end = current_segment_start + timedelta(minutes=10) - timedelta(microseconds=1)
    recent_10min_start_minute = current_segment_start_minute - 10
```

**文件2：`run_sync.bat`**

无需修改（仍然调用 `data_sync.py --realtime`）

**文件3：Windows 任务计划程序**

需要修改任务的执行频率：
- 从：每5分钟执行一次
- 改为：每10分钟执行一次

### 2. 改为覆盖式写入

#### 需要修改的文件

**文件：`data_sync.py` - `sync_realtime_data_ten_minutes()` 函数**

**关键代码修改：**

```python
# 原代码（累加式）：
db.insert_or_update_hourly_data_incremental(
    time_hour, total_sales, total_orders, total_visitors
)

# 改为（覆盖式）：
# 1. 查询该小时已有的数据
existing_data = db.get_hourly_data_by_time(time_hour)
if existing_data:
    # 累加：已有数据 + 新数据
    total_sales = existing_data['total_gmv'] + total_sales
    total_orders = existing_data['total_orders'] + total_orders
    # 访客数：使用新收集的值（因为访客数是按天去重的）
    total_visitors = total_visitors  # 使用新值
else:
    # 如果没有已有数据，直接使用新数据（首次写入）
    pass  # total_sales, total_orders 已经是新数据

# 2. 覆盖写入
avg_order_value = total_sales / total_orders if total_orders > 0 else 0.0
db.insert_or_update_hourly_data(
    time_hour, total_sales, total_orders, total_visitors, avg_order_value
)
```

**单店铺明细表同样需要改为覆盖式：**

```python
# 原代码（累加式）：
db.insert_or_update_store_hourly_incremental(
    shop_domain, time_hour, data['sales'], data['orders'], data['visitors']
)

# 改为（覆盖式）：
# 1. 查询单店铺已有数据
existing_store_data = db.get_store_hourly_data(
    shop_domain, time_hour, time_hour + timedelta(hours=1) - timedelta(seconds=1)
)
if existing_store_data and len(existing_store_data) > 0:
    existing = existing_store_data[0]
    store_total_sales = existing['total_gmv'] + data['sales']
    store_total_orders = existing['total_orders'] + data['orders']
    store_total_visitors = data['visitors']  # 使用新值
else:
    store_total_sales = data['sales']
    store_total_orders = data['orders']
    store_total_visitors = data['visitors']

# 2. 覆盖写入（需要获取owner信息）
owner = store.get('owner', '')  # 需要从store对象获取
db.insert_or_update_store_hourly(
    shop_domain, time_hour, store_total_sales, store_total_orders, 
    store_total_visitors
)
```

## ✅ 优势分析

### 1. 彻底解决数据翻倍问题

**双重保障机制：**

1. **降低执行频率**：
   - 从每5分钟改为每10分钟
   - 进程重叠的概率大大降低
   - 即使执行时间稍长（例如6-7分钟），也不会与下次执行重叠

2. **覆盖式写入**：
   - 即使两个进程同时执行，覆盖写入也不会导致数据翻倍
   - 因为每次写入都是完整的小时数据（已有数据 + 新数据）

### 2. 降低系统负载

**资源消耗对比：**

| 资源类型 | 5分钟间隔 | 10分钟间隔 | 节省 |
|---------|----------|-----------|------|
| **执行次数** | 288次/天 | 144次/天 | **50%** |
| **API调用** | 约10,080次/天（35店铺×288次） | 约5,040次/天 | **50%** |
| **数据库写入** | 约10,080次/天 | 约5,040次/天 | **50%** |
| **系统资源** | 高 | 中 | **降低** |

### 3. 提升数据准确性

- **覆盖式写入**：每次写入都是完整的小时数据，易于验证和调试
- **降低竞态条件**：执行频率降低，进程重叠概率大幅降低
- **数据一致性**：即使出现异常情况，覆盖写入也能保证数据的最终一致性

### 4. 易于维护和调试

- 数据更清晰：每次写入都是完整的小时数据
- 问题定位更容易：如果数据有问题，可以直接查看数据库中的数据
- 日志更简洁：执行频率降低，日志量减少50%

---

## ⚠️ 需要注意的问题

### 1. 数据延迟增加

- **当前**：约5-10分钟的数据延迟
- **优化后**：约10-20分钟的数据延迟
- **影响评估**：对于大多数业务场景，10-20分钟的数据延迟是可以接受的

### 2. 单店铺明细表的owner字段

- **问题**：覆盖式写入需要owner字段，但当前代码中可能没有直接获取
- **解决方案**：需要从店铺信息中获取owner，或者在查询时一起获取

### 3. 其他脚本的兼容性

- **`fill_today_data.py`**：仍然使用5分钟段，需要保持一致
- **`fill_date_data.py`**：仍然使用5分钟段，需要保持一致
- **`fill_missing_data.py`**：仍然使用5分钟段，需要保持一致
- **注意**：这些补全脚本仍然使用5分钟段是合理的，因为它们用于历史数据补全，不是实时同步

### 4. 跨天处理逻辑

- **当前**：00:05:00 开始收集今天的数据
- **优化后**：00:10:00 开始收集今天的数据
- **影响**：跨天后的第一个10分钟段会在00:10:00开始收集
