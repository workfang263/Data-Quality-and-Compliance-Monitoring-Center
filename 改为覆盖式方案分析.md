# 改为覆盖式写入方案分析

## 当前情况

### 店铺数量
- **启用的店铺**：**35个**（已确认）
- **总店铺数**：约36个（包括已禁用的）

### 当前写入模式：累加式（Incremental）

**工作流程：**
```
每5分钟执行一次
  ├─ 收集最近一个5分钟段的数据（所有店铺）
  ├─ 按小时汇总（例如：10:05-10:09 → 10:00小时）
  └─ 使用 insert_or_update_hourly_data_incremental() 累加写入
     └─ SQL: total_gmv = total_gmv + VALUES(total_gmv)  (累加)
```

**优点：**
- 适合实时同步：每次只收集5分钟的数据
- 不需要等待整个小时完成
- 数据可以逐步累加

**缺点：**
- 容易产生竞态条件（导致数据翻倍）
- 如果重复执行，数据会重复累加

---

## 改为覆盖式方案分析

### 方案1：按小时收集，覆盖写入 ✅ 推荐

**工作流程：**
```
每5分钟执行一次
  ├─ 收集最近一个5分钟段的数据（所有店铺）
  ├─ 查询该小时已收集的所有数据（从数据库或内存）
  ├─ 累加计算该小时的完整数据
  └─ 使用 insert_or_update_hourly_data() 覆盖写入
     └─ SQL: total_gmv = VALUES(total_gmv)  (覆盖)
```

**优点：**
- ✅ 避免竞态条件：即使重复执行，覆盖写入不会导致数据翻倍
- ✅ 数据更准确：每次写入都是该小时的完整数据
- ✅ 更容易调试：可以清楚看到每个小时的数据

**缺点：**
- ⚠️ 需要维护每个小时的累加数据（在内存中或临时存储）
- ⚠️ 逻辑稍复杂：需要先查询已有数据，再累加，再覆盖

**技术实现：**
1. 收集5分钟段数据后，先查询该小时已有的数据
2. 在内存中累加：新数据 + 已有数据
3. 覆盖写入累加后的完整数据

---

### 方案2：改为每小时执行一次 ❌ 不推荐

**工作流程：**
```
每小时执行一次（例如：10:00执行）
  ├─ 收集上一个完整小时的数据（例如：09:00-09:59:59）
  └─ 覆盖写入
```

**缺点：**
- ❌ 数据延迟：要等整小时完成才能收集
- ❌ 如果执行失败，会丢失整小时数据
- ❌ 用户体验差：数据更新不及时

---

## 可行性评估

### ✅ 可操作性：高

**理由：**
1. 数据库已有覆盖式的函数：`insert_or_update_hourly_data()`
2. 逻辑改动不大：只需要在累加后再覆盖写入
3. 不会影响现有数据结构

### ⚠️ 需要注意的问题

1. **数据累加逻辑**
   - 需要从数据库查询该小时已有的数据
   - 在内存中累加新数据和已有数据
   - 然后覆盖写入

2. **性能考虑**
   - 每次写入前需要查询一次数据库
   - **35个店铺**，每个小时查询一次汇总表数据，性能影响可接受（查询很快）
   - 单店铺明细表需要查询35次（每个店铺一次），可以优化为批量查询

3. **边界情况**
   - 如果查询失败怎么办？
   - 如果某个5分钟段的数据没有成功写入，下次如何处理？

## 具体实现方案

### 修改点1：`sync_realtime_data_five_minutes()` 函数

**当前逻辑：**
```python
# 收集5分钟段数据
total_sales = ...  # 累加所有店铺的销售额
total_orders = ...  # 累加所有订单数

# 直接累加写入
db.insert_or_update_hourly_data_incremental(
    time_hour, total_sales, total_orders, total_visitors
)
```

**改为覆盖式：**
```python
# 收集5分钟段数据
new_sales = ...  # 本次收集的销售额
new_orders = ...  # 本次收集的订单数

# 查询该小时已有的数据
existing_data = db.get_hourly_data_by_time(time_hour)
if existing_data:
    # 累加：已有数据 + 新数据
    total_sales = existing_data['total_gmv'] + new_sales
    total_orders = existing_data['total_orders'] + new_orders
    # 访客数：使用最新值（因为访客数是按天去重的）
    total_visitors = total_visitors  # 使用新收集的值
else:
    # 如果没有已有数据，直接使用新数据
    total_sales = new_sales
    total_orders = new_orders

# 覆盖写入
db.insert_or_update_hourly_data(
    time_hour, total_sales, total_orders, total_visitors, 
    avg_order_value=total_sales/total_orders if total_orders > 0 else 0
)
```

### 修改点2：单店铺明细表

**同样需要改为覆盖式：**
```python
# 查询单店铺该小时已有的数据
existing_store_data = db.get_store_hourly_data_by_time(shop_domain, time_hour)
if existing_store_data:
    # 累加
    total_sales = existing_store_data['total_gmv'] + new_sales
    total_orders = existing_store_data['total_orders'] + new_orders
else:
    total_sales = new_sales
    total_orders = new_orders

# 覆盖写入
db.insert_or_update_store_hourly(...)
```

## 风险评估

### ✅ 优点
1. **解决竞态条件**：覆盖写入不会导致数据翻倍
2. **数据更准确**：每次写入都是完整的小时数据
3. **易于调试**：可以清楚看到每个小时的数据

### ⚠️ 风险
1. **性能影响**：每次写入前需要查询一次数据库（38个店铺，可接受）
2. **逻辑复杂度**：需要先查询，再累加，再覆盖（逻辑稍复杂）
3. **边界情况**：如果查询失败，需要fallback到累加模式

---

## 建议

### 方案A：改为覆盖式 ✅ 推荐

**优点：**
- 彻底解决竞态条件问题
- 数据更准确，易于调试

**缺点：**
- 需要修改代码逻辑
- 性能略有影响（每次查询一次）

### 方案B：保持累加式 + 添加进程锁

**优点：**
- 代码改动最小
- 性能不受影响

**缺点：**
- 需要添加进程锁机制
- 如果进程锁失效，仍有数据翻倍风险

---

## 结论

**建议采用方案A（改为覆盖式）**，理由：
1. 从根本上解决竞态条件问题（即使重复执行，覆盖写入不会导致数据翻倍）
2. **35个店铺**的数据量，查询性能影响可接受（汇总表查询很快）
3. 代码改动不大，风险可控（数据库已有覆盖式函数）
4. 数据更准确，易于调试和维护

**关键优势：**
- ✅ 覆盖式写入避免了累加模式导致的重复累加问题
- ✅ 即使两个进程同时执行，最后写入的数据也是正确的（因为都基于相同的已有数据累加）
- ✅ 35个店铺的查询量在可接受范围内
