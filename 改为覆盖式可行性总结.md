# 改为覆盖式写入可行性总结

## 当前情况

- **启用的店铺数量**：**35个**
- **当前写入模式**：累加式（Incremental）
- **问题**：竞态条件导致数据翻倍

---

## 改为覆盖式的可操作性评估

### ✅ **可操作性：高（强烈推荐）**

### 理由：

1. **数据库支持完善**
   - ✅ 已有覆盖式函数：`insert_or_update_hourly_data()`（覆盖模式）
   - ✅ 已有查询函数：`get_hourly_data_by_time()`（查询单个小时数据）
   - ✅ 单店铺数据可以用 `get_store_hourly_data()` 查询

2. **代码改动量适中**
   - 主要修改：`sync_realtime_data_five_minutes()` 函数
   - 需要修改两个地方：
     - 汇总表写入（1处）
     - 单店铺明细表写入（35个店铺，但逻辑相同）

3. **性能影响可接受**
   - 汇总表：每次写入前查询1次（很快）
   - 单店铺明细表：需要查询35次（可以优化为批量查询，或使用现有的查询方法）
   - 35个店铺的数据量，性能影响在可接受范围内

4. **能解决根本问题**
   - ✅ **彻底解决竞态条件**：即使两个进程同时执行，覆盖写入也不会导致数据翻倍
   - ✅ **数据更准确**：每次写入都是完整的小时数据
   - ✅ **易于调试**：可以清楚看到每个小时的数据

## 技术实现方案

### 核心逻辑

```
每5分钟执行一次：
1. 收集最近5分钟段的数据（新数据）
2. 查询该小时已有的数据（从数据库）
3. 累加：已有数据 + 新数据 = 完整小时数据
4. 覆盖写入：使用 insert_or_update_hourly_data() 覆盖写入
```

### 关键代码修改

**汇总表写入（data_sync.py:1764-1766）：**

```python
# 当前代码（累加式）：
db.insert_or_update_hourly_data_incremental(
    time_hour, total_sales, total_orders, total_visitors
)

# 改为覆盖式：
# 1. 查询已有数据
existing_data = db.get_hourly_data_by_time(time_hour)
if existing_data:
    # 累加：已有数据 + 新数据
    total_sales = existing_data['total_gmv'] + total_sales
    total_orders = existing_data['total_orders'] + total_orders
    # 访客数：使用新收集的值（因为访客数是按天去重的，应该使用最新值）
    total_visitors = total_visitors  # 使用新值
else:
    # 如果没有已有数据，直接使用新数据（首次写入）
    pass  # total_sales, total_orders 已经是新数据

# 2. 覆盖写入
avg_order_value = total_sales / total_orders if total_orders > 0 else 0.0
db.insert_or_update_hourly_data(
    time_hour, total_sales, total_orders, total_visitors, avg_order_value
)
```

## 优点

1. ✅ **彻底解决竞态条件**
   - 即使两个进程同时执行，覆盖写入也不会导致数据翻倍
   - 因为每次写入都是完整的小时数据（已有数据+新数据）

2. ✅ **数据更准确**
   - 每次写入都是完整的小时数据
   - 不会因为重复执行而累加错误的数据

3. ✅ **易于调试和维护**
   - 可以清楚看到每个小时的数据
   - 如果数据有问题，更容易定位

4. ✅ **性能影响可接受**
   - 35个店铺，查询性能影响很小
   - 可以进一步优化（批量查询、缓存等）

## 潜在问题和解决方案

### 问题1：查询失败怎么办？

**解决方案：**
- 如果查询失败，可以fallback到累加模式
- 或者记录日志，下次重试

### 问题2：两个进程同时查询和写入？

**分析：**
- 即使两个进程同时查询到相同的已有数据
- 都累加新数据后覆盖写入
- 由于都是基于相同的已有数据累加，最终结果也是正确的
- **关键是覆盖写入，而不是累加写入**

### 问题3：单店铺明细表的owner字段

**解决方案：**
- 需要从店铺信息中获取owner
- 或者在查询时一起获取
