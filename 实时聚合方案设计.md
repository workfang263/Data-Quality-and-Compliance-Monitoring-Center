# 实时聚合方案设计

## 📋 需求分析

**目标**：实现类似 MySQL 触发器的实时聚合机制
- 当 `shoplazza_store_hourly` 或 `fb_ad_account_spend_hourly` 数据更新时
- 自动触发聚合任务，更新 `owner_daily_summary` 表
- 要求实时性好，不影响主流程

---

## 🔍 数据写入点分析

### 写入点1：店铺数据同步
- **位置**：`database.py`
  - `insert_or_update_store_hourly()` - 覆盖模式（历史同步）
  - `insert_or_update_store_hourly_incremental()` - 增量模式（5分钟实时同步）
- **调用频率**：每5分钟一次，可能写入多个店铺、多个小时的数据

### 写入点2：广告花费同步
- **位置**：`fb_spend_sync.py`
  - `upsert_spend()` - 写入当天数据
- **调用频率**：每5分钟一次，写入36个账户的数据

### 写入点3：广告花费回填
- **位置**：`fb_spend_backfill.py`
  - `upsert_spend()` - 批量写入历史数据
- **调用频率**：一次性任务，批量写入

---

## 💡 方案对比

### 方案1：同步调用聚合函数（最简单）

**实现方式**：
- 在数据写入函数中，写入完成后立即调用聚合函数
- 直接调用 `aggregate_owner_daily.py` 中的聚合函数

**代码示例**：
```python
# 在 database.py 的写入函数中
def insert_or_update_store_hourly_incremental(...):
    # ... 写入数据 ...
    conn.commit()
    
    # 立即聚合
    from aggregate_owner_daily import aggregate_date
    aggregate_date(conn, time_hour.date(), verbose=False)
    
    return True
```

**优点**：
- ✅ 实现简单，代码改动小
- ✅ 实时性好，数据写入后立即聚合
- ✅ 不需要额外的服务或组件

**缺点**：
- ❌ 如果聚合失败，可能影响主流程（但可以捕获异常）
- ❌ 如果同一时间写入多个小时的数据，会重复聚合（性能浪费）
- ❌ 聚合是同步的，会稍微延长写入时间（通常<1秒）

**适用场景**：写入频率不高，对实时性要求高

---

### 方案2：异步调用聚合函数（推荐⭐）

**实现方式**：
- 在数据写入函数中，写入完成后启动后台线程执行聚合
- 使用 `threading.Thread` 或 `concurrent.futures.ThreadPoolExecutor`
- 聚合失败不影响主流程

**代码示例**：
```python
import threading
from aggregate_owner_daily import aggregate_date

# 全局聚合线程池（单例）
_aggregate_executor = None

def get_aggregate_executor():
    global _aggregate_executor
    if _aggregate_executor is None:
        from concurrent.futures import ThreadPoolExecutor
        _aggregate_executor = ThreadPoolExecutor(max_workers=1, thread_name_prefix="aggregate")
    return _aggregate_executor

# 在 database.py 的写入函数中
def insert_or_update_store_hourly_incremental(...):
    # ... 写入数据 ...
    conn.commit()
    
    # 异步聚合（不阻塞）
    target_date = time_hour.date()
    executor = get_aggregate_executor()
    executor.submit(_aggregate_date_async, target_date)
    
    return True

def _aggregate_date_async(target_date):
    """后台线程执行聚合"""
    try:
        conn = get_db_conn()
        aggregate_date(conn, target_date, verbose=False)
        conn.close()
    except Exception as e:
        logger.error(f"聚合失败 {target_date}: {e}")
```

**优点**：
- ✅ 实时性好，数据写入后立即触发聚合
- ✅ 不阻塞主流程，聚合失败不影响数据写入
- ✅ 可以控制并发（单线程执行，避免重复聚合）

**缺点**：
- ⚠️ 需要管理线程池（但实现简单）
- ⚠️ 如果同一时间写入多个小时的数据，可能重复聚合（可以通过去重优化）

**适用场景**：推荐方案，平衡了实时性和稳定性

### 方案3：批量聚合（性能最优）

**实现方式**：
- 在数据写入时，记录需要聚合的日期到集合中
- 在同步脚本的最后，统一聚合所有记录的日期
- 使用 `set` 去重，避免重复聚合

**代码示例**：
```python
# 在 data_sync.py 中
_aggregate_dates = set()  # 需要聚合的日期集合

def insert_or_update_store_hourly_incremental(...):
    # ... 写入数据 ...
    conn.commit()
    
    # 记录需要聚合的日期
    _aggregate_dates.add(time_hour.date())
    
    return True

# 在同步脚本的最后
def sync_realtime_data_five_minutes():
    # ... 同步数据 ...
    
    # 统一聚合所有日期
    if _aggregate_dates:
        from aggregate_owner_daily import aggregate_date
        conn = get_db_conn()
        for date in _aggregate_dates:
            aggregate_date(conn, date, verbose=False)
        conn.close()
        _aggregate_dates.clear()
```

**优点**：
- ✅ 性能最优，减少聚合次数（同一日期只聚合一次）
- ✅ 不阻塞主流程
- ✅ 适合批量写入场景

**缺点**：
- ⚠️ 需要修改多个文件（data_sync.py, fb_spend_sync.py）
- ⚠️ 需要管理全局状态（日期集合）
- ⚠️ 如果脚本异常退出，可能丢失聚合（但可以最后统一聚合）

**适用场景**：写入频率高，对性能要求高

---

### 方案4：数据库触发器 + 外部脚本（不推荐）

**实现方式**：
- 在 MySQL 中创建触发器，数据更新时写入标记表
- 外部脚本轮询标记表，执行聚合

**优点**：
- ✅ 数据库层面自动触发
- ✅ 完全解耦

**缺点**：
- ❌ MySQL 触发器不能直接调用 Python 脚本
- ❌ 需要额外的轮询脚本
- ❌ 架构复杂，维护成本高
- ❌ 触发器可能影响写入性能

**适用场景**：不推荐，过于复杂

---

### 方案5：消息队列 + 后台服务（过度设计）

**实现方式**：
- 数据写入后发送消息到 Redis/RabbitMQ
- 后台服务监听消息，执行聚合

**优点**：
- ✅ 完全解耦
- ✅ 可以横向扩展

**缺点**：
- ❌ 需要额外的消息队列服务
- ❌ 架构复杂，维护成本高
- ❌ 对于当前场景，过度设计

**适用场景**：大规模分布式系统，不适用于当前场景

## 🎯 推荐方案：方案2（异步调用）+ 方案3（批量优化）混合

### 设计思路

**核心策略**：
1. **异步执行**：使用后台线程执行聚合，不阻塞主流程
2. **批量去重**：收集需要聚合的日期，最后统一聚合，避免重复
3. **错误隔离**：聚合失败不影响数据写入，只记录日志

### 实现框架

#### 1. 创建聚合管理器（`aggregate_manager.py`）

```python
"""
聚合管理器：统一管理实时聚合任务
- 收集需要聚合的日期
- 异步执行聚合
- 批量去重，避免重复聚合
"""
import threading
from datetime import date
from typing import Set
from concurrent.futures import ThreadPoolExecutor
import logging

logger = logging.getLogger(__name__)

class AggregateManager:
    """聚合管理器（单例）"""
    _instance = None
    _lock = threading.Lock()
    
    def __new__(cls):
        if cls._instance is None:
            with cls._lock:
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
                    cls._instance._initialized = False
        return cls._instance
    
    def __init__(self):
        if self._initialized:
            return
        self._pending_dates: Set[date] = set()
        self._date_lock = threading.Lock()
        self._executor = ThreadPoolExecutor(max_workers=1, thread_name_prefix="aggregate")
        self._initialized = True
    
    def schedule_aggregate(self, target_date: date):
        """
        调度聚合任务（非阻塞）
        如果同一日期已经在队列中，不会重复添加
        """
        with self._date_lock:
            if target_date not in self._pending_dates:
                self._pending_dates.add(target_date)
                # 异步执行聚合
                self._executor.submit(self._aggregate_date_async, target_date)
    
    def _aggregate_date_async(self, target_date: date):
        """后台线程执行聚合"""
        try:
            from aggregate_owner_daily import aggregate_date
            from config import DB_CONFIG
            import pymysql
            
            conn = pymysql.connect(
                host=DB_CONFIG["host"],
                port=DB_CONFIG["port"],
                user=DB_CONFIG["user"],
                password=DB_CONFIG["password"],
                database=DB_CONFIG["database"],
                charset=DB_CONFIG.get("charset", "utf8mb4"),
                cursorclass=pymysql.cursors.DictCursor,
                autocommit=False,
            )
            
            aggregate_date(conn, target_date, verbose=False)
            conn.close()
            
            # 从待处理集合中移除
            with self._date_lock:
                self._pending_dates.discard(target_date)
                
        except Exception as e:
            logger.error(f"聚合失败 {target_date}: {e}", exc_info=True)
            # 从待处理集合中移除（避免重复失败）
            with self._date_lock:
                self._pending_dates.discard(target_date)
    
    def get_pending_count(self) -> int:
        """获取待处理的日期数量（用于监控）"""
        with self._date_lock:
            return len(self._pending_dates)

# 全局单例
aggregate_manager = AggregateManager()
```

## 🎯 最终推荐：混合方案（方案2 + 方案3）

### 为什么选择混合方案？

1. **实时性好**：数据写入后立即触发聚合（异步）
2. **性能最优**：批量去重，同一日期只聚合一次
3. **稳定性高**：聚合失败不影响主流程
4. **实现简单**：只需要创建一个管理器类，修改少量代码

## ⚠️ 注意事项

1. **并发控制**：使用单线程执行器，避免并发聚合同一日期
2. **错误处理**：聚合失败只记录日志，不影响主流程
3. **资源管理**：聚合函数需要创建新的数据库连接（避免连接冲突）
4. **去重机制**：使用 `set` 去重，避免重复聚合
