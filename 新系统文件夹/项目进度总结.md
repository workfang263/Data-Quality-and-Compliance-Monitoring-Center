# 项目进度总结

## 📋 项目概述

**项目名称**：Streamlit → Vue + FastAPI 迁移项目

**项目目标**：将现有的 Streamlit 系统迁移到 Vue + FastAPI 架构，实现前后端分离。

**核心原则**：
1. ✅ **完全隔离**：新旧系统互不影响（方案二：复制代码）
2. ✅ **并行运行**：两个系统同时运行，不同端口
3. ✅ **共用数据库**：数据一致，无需迁移
4. ✅ **逐步切换**：功能模块逐步迁移，测试通过后再切换

**技术栈**：
- **后端**：FastAPI 0.104.1 + Uvicorn + PyMySQL + Pydantic
- **前端**：Vue 3.3.13 + TypeScript + Vite + Element Plus + ECharts
- **数据库**：MySQL（共用旧系统数据库）

## 🐛 遇到的问题和解决方案

### 窗口1：项目初始化和后端搭建

#### 问题1：uvicorn未安装
**现象**：`uvicorn : 无法将"uvicorn"项识别为 cmdlet、函数、脚本文件或可运行程序的名称`

**原因**：未安装uvicorn或不在PATH中

**解决方案**：
```bash
cd backend
pip install -r requirements.txt
```

---

#### 问题2：requirements.txt中文注释导致编码错误
**现象**：`UnicodeDecodeError: 'gbk' codec can't decode byte 0xab in position 28`

**原因**：requirements.txt包含中文注释，Windows默认使用GBK编码读取

**解决方案**：移除requirements.txt中的中文注释，改为英文注释

---

#### 问题3：PowerShell不支持&&
**现象**：`PowerShell 不支持 &&`

**原因**：PowerShell使用`;`而不是`&&`来连接命令

**解决方案**：将命令拆分为多个独立的命令执行

---

#### 问题4：路径错误
**现象**：`cd : 找不到路径"C:\Users\EDY\backend"，因为它不存在`

**原因**：在错误的目录执行cd命令

**解决方案**：先切换到项目根目录，再进入backend目录
```bash
cd "d:\projects\line chart"
cd backend
```

---

#### 问题5：端口占用
**现象**：`[WinError 10013] 以一种访问权限不允许的方式做了一个访问套接字的尝试`

**原因**：端口8000或8001被其他进程占用

**解决方案**：
1. 查找占用端口的进程：`netstat -ano | findstr :8001`
2. 终止进程：`taskkill /PID <PID> /F`
3. 或更换端口：`uvicorn app.main:app --reload --port 8002`

**最终方案**：后端服务运行在8002端口

---

#### 问题6：404 Not Found
**现象**：访问`http://localhost:8000/`或`http://localhost:8000/docs`返回404

**原因**：
1. Python包导入问题：缺少`__init__.py`文件
2. 导入路径错误：使用了`sys.path.insert`而不是标准相对导入

**解决方案**：
1. 创建`__init__.py`文件：
   - `backend/app/__init__.py`
   - `backend/app/api/__init__.py`
   - `backend/app/services/__init__.py`
   - `backend/app/models/__init__.py`
2. 修改导入语句：从`sys.path.insert`改为标准相对导入`from app.services.database_new import Database`

### 窗口2：前端搭建和看板页面实现

#### 问题7：Axios类型导入错误
**现象**：`[Vue Router warn]: uncaught error during route navigation: SyntaxError: The requested module '/node_modules/.vite/deps/axios.js?v=25c8a5c5' does not provide an export named 'AxiosInstance'`

**原因**：
1. `AxiosInstance`等是类型，应该使用`import type`导入
2. `AxiosRequestConfig`在Axios 1.x中已改为`InternalAxiosRequestConfig`

**解决方案**：
```typescript
// 修改前
import axios, { AxiosInstance, AxiosRequestConfig, AxiosResponse, AxiosError } from 'axios'

// 修改后
import axios from 'axios'
import type { AxiosInstance, InternalAxiosRequestConfig, AxiosResponse, AxiosError } from 'axios'
```

#### 问题9：负责人汇总页面404错误
**现象**：访问负责人汇总页面时，API请求返回404

**原因**：
1. `frontend/src/api/owners.ts`中错误地使用了`.then()`链处理响应，但`request.ts`的拦截器已经处理了响应
2. 后端服务未重启，新的`owners_api`路由未加载

**解决方案**：
1. 移除`owners.ts`中的`.then()`链，直接返回`request.get()`的结果
2. 重启后端服务以加载新的路由

#### 问题10：后端服务无法访问
**现象**：`http://localhost:8001/docs`和`http://localhost:8001/`无法访问，页面一直转圈

**原因**：`Database()`初始化在模块级别执行，可能导致阻塞操作

**解决方案**：
1. 将`Database`初始化改为延迟加载（使用`Depends(get_db)`）
2. 在`auth_api.py`中实现单例模式的`get_db()`函数：
   ```python
   db_instance: Optional[Database] = None

   def get_db():
       """获取数据库实例（单例模式）"""
       global db_instance
       if db_instance is None:
           db_instance = Database()
       return db_instance
   ```
3. 在路由函数中使用依赖注入：`db: Database = Depends(get_db)`

### 窗口4：数据准确性和访客数计算问题

#### 问题15：看板总访客数与负责人汇总不一致
**现象**：看板显示总访客数72,238，但负责人汇总的访客数总和只有47,527

**原因**：
1. 天粒度数据：`get_daily_data_with_spend`没有添加映射表过滤，包含了未映射的旧数据
2. 小时粒度数据：`get_hourly_data_with_spend`从`shoplazza_store_hourly`查询，但历史数据的`owner`字段可能为NULL

**解决方案**：
1. **天粒度修复**：在`get_daily_data_with_spend`中添加映射表过滤（`store_owner_mapping`、`ad_account_owner_mapping`、`tt_ad_account_owner_mapping`）
2. **前端访客数计算优化**：在`DashboardSummary.vue`中添加按天去重逻辑，小时粒度按天分组取最大值后累加

**相关文件**：
- `backend/app/services/database_new.py` - `get_daily_data_with_spend`方法
- `frontend/src/components/DashboardSummary.vue` - 访客数计算逻辑

#### 问题16：小时粒度无数据显示
**现象**：选择"小时"颗粒度时，看板显示"该时间段内没有数据"

**原因**：
1. `get_hourly_data_with_spend`从`shoplazza_store_hourly`查询，添加了映射表过滤
2. 历史数据的`owner`字段可能为NULL，导致过滤后没有数据
3. 尝试使用JOIN映射表，但遇到字符集排序规则冲突

**解决方案**：
1. **回退到旧系统逻辑**：使用`shoplazza_overview_hourly`汇总表（与旧系统一致）
2. **移除映射表过滤**：完全对齐旧系统，不加映射表过滤
3. **后端预处理访客数**：在`dashboard_api.py`中添加预处理步骤，确保同一天所有小时的访客数相同（取最大值）

**相关文件**：
- `backend/app/services/database_new.py` - `get_hourly_data_with_spend`方法
- `backend/app/api/dashboard_api.py` - 数据预处理逻辑
- `frontend/src/components/DashboardSummary.vue` - 访客数计算逻辑

---

#### 问题17：字符集排序规则冲突
**现象**：`pymysql.err.OperationalError: (1267, "Illegal mix of collations (utf8mb4_unicode_ci,IMPLICIT) and (utf8mb4_0900_ai_ci,IMPLICIT) for operation '='")`

**原因**：`store_owner_mapping`表的`shop_domain`字段使用`utf8mb4_unicode_ci`，`shoplazza_store_hourly`表的`shop_domain`字段使用`utf8mb4_0900_ai_ci`，JOIN时排序规则不一致

**解决方案**：在JOIN条件中添加`COLLATE utf8mb4_unicode_ci`统一排序规则（但最终采用回退到`shoplazza_overview_hourly`的方案，避免了这个问题）

---

#### 问题18：访客数计算不准确（小时粒度）
**现象**：小时粒度下，看板总访客数显示72,238，与旧系统（46,518）不一致

**原因**：
1. 后端没有预处理数据，确保同一天所有小时的访客数相同
2. 前端计算逻辑可能有问题，或者数据库返回的数据本身就不一致

**解决方案**：
1. **后端预处理**：在`dashboard_api.py`中，小时粒度数据先按天分组取最大访客数，然后更新所有小时的访客数为该天的最大值
2. **前端计算优化**：在`DashboardSummary.vue`中，添加`granularity`参数，准确判断数据粒度，小时粒度按天分组取最大值后累加

**相关文件**：
- `backend/app/api/dashboard_api.py` - 数据预处理逻辑
- `frontend/src/components/DashboardSummary.vue` - 访客数计算逻辑
- `frontend/src/views/Dashboard.vue` - 传递`granularity`参数

---

#### 问题19：数据收集逻辑对齐
**现象**：新系统与旧系统使用同一个数据库，但访客数不一致

**原因**：新系统尝试添加映射表过滤，但旧系统没有过滤，导致数据不一致

**解决方案**：
1. **完全对齐旧系统逻辑**：

2. **修复前端日期解析**：
   - 直接提取ISO字符串的日期部分，避免时区转换
   - 使用`item.time_hour.split('T')[0]`直接提取日期，而不是通过`new Date().toISOString()`

## 📝 访客数收集逻辑说明（重要）

### 访客数的特殊性

**重要特性**：
- 访客数是**累计值**（不是增量值）：表示00:00:00到当前小时的累计访客数
- **覆盖更新**：每5分钟同步一次，收集当天00:00:00到当前时刻的累计访客数，然后**覆盖**对应小时的数据（不是累加）
- **同一天内递增**：同一天内不同小时是递增的（00:00 ≤ 01:00 ≤ ... ≤ 23:00）

**数据收集逻辑示例**：
- 在15:05:00运行脚本，收集00:00:00-15:05:00的访客数，然后覆盖15:00:00的数据
- 15:10:00运行脚本，收集00:00:00-15:10:00的访客数，再次覆盖15:00:00的数据（更新为更大值）

**计算总访客数的正确方法**：
- **小时粒度**：按天分组取最大值（即当天的总访客数），然后累加所有天的最大值（不同天的访客应该累加）
- **天粒度**：直接累加（每天的数据已经是当天的总访客数）

**注意事项**：
- ⚠️ **使用MAX而不是SUM**：因为访客数是累计值，不应该累加，应该取最大值
- ⚠️ **避免时区转换**：解析ISO日期字符串时，直接提取日期部分，避免`new Date().toISOString()`的时区转换问题
