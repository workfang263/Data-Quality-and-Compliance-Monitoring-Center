## 📋 需求概述

### 用户角色

1. **主管理员（admin）**：
   - 权限不变，可以查看所有数据
   - 新增功能：可以给普通用户授权，允许查看哪些负责人的数据
   - 可以勾选多个负责人授权给普通用户

2. **普通用户（user）**：
   - 只能查看被授权的负责人的数据和折线图
   - 默认不能查看总数据和总数据折线图（除非授予权限）
   - 默认不能编辑映射（除非授予权限）
   - 左侧菜单栏功能开放，但数据权限受限

### 功能需求

1. **注册功能**：
   - 新增注册页面
   - 只需要账户、密码、确认密码
   - 注册成功后为普通用户，没有授权什么都看不了

2. **权限管理功能**：
   - 单独开一个页面（只有admin可以访问）
   - 管理员可以给普通用户授权：
     - 勾选多个负责人（负责人权限）
     - 勾选"允许查看看板总数据"（扩展权限）
     - 勾选"允许编辑映射"（扩展权限）

## 🗄️ 数据库设计

### 1. 用户表（users）- 修改

**需要修改**：
- `role` 字段：`enum('admin','user')`
- 添加扩展权限字段：
  - `can_view_dashboard` - 是否可以查看看板总数据（默认FALSE）
  - `can_edit_mappings` - 是否可以编辑映射（默认FALSE）

**SQL脚本**：`添加用户扩展权限字段.sql`

### 2. 用户权限表（user_owner_permissions）- 新建

**表结构**：
```sql
CREATE TABLE user_owner_permissions (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    owner VARCHAR(100) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY unique_user_owner (user_id, owner),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

**SQL脚本**：`创建用户权限表.sql`

## 🔐 权限类型说明

### 1. 负责人权限

**说明**：控制普通用户可以查看哪些负责人的数据

**存储位置**：`user_owner_permissions` 表

**控制效果**：
- 负责人汇总页面：只显示被授权的负责人
- 看板页面：如果有 `can_view_dashboard` 权限，只显示被授权负责人的聚合总数据
- 负责人小时数据：只能查看被授权的负责人的数据

### 2. 查看看板总数据权限（`can_view_dashboard`）

**说明**：控制普通用户是否可以查看看板总数据

**存储位置**：`users.can_view_dashboard` 字段

**控制效果**：
- 如果没有权限：看板页面显示提示，隐藏总数据
- 如果有权限：显示被授权负责人的聚合总数据（不是所有负责人的数据）

### 3. 编辑映射权限（`can_edit_mappings`）

**说明**：控制普通用户是否可以编辑映射

**存储位置**：`users.can_edit_mappings` 字段

**控制效果**：
- 如果没有权限：映射编辑页面显示提示，禁用所有编辑功能
- 如果有权限：可以正常编辑映射

### 4. 管理员权限

**说明**：管理员（role='admin'）自动拥有所有权限

**控制效果**：
- 可以查看所有数据
- 可以编辑所有映射
- 可以管理用户权限

## 🐛 问题记录

### 问题1：看板小时粒度数据缺失（2025-12-22）

**现象**：普通用户看板页面小时粒度数据不显示

**原因**：JOIN字符集冲突，`get_hourly_data_with_spend_filtered`方法使用`INNER JOIN store_owner_mapping`过滤

**解决方案**：移除JOIN，直接使用`owner`字段过滤

**详细说明**：见 `数据准确性修复完整文档.md`

### 问题2：小时粒度访客数计算不准确（2025-12-22）

**现象**：天粒度访客数4968（正确），小时粒度访客数3398（错误）

**原因**：只取了某个店铺当天的最大访客数，没有累加所有店铺的访客数

**解决方案**：后端预处理，先按店铺取MAX，再累加所有店铺和负责人的访客数

**详细说明**：见 `数据准确性修复完整文档.md`
