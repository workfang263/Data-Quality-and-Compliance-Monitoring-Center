## 问题1：普通用户看板小时粒度数据缺失

### 📋 问题描述

**现象**：
- 普通用户test被授予了三个负责人权限（21、Lucky、Williams）和"允许查看看板总数据"权限
- 负责人汇总页面：正常显示三个负责人的数据 ✅
- 看板页面：
  - 天粒度数据：正常显示 ✅
  - 小时粒度数据：没有数据显示 ❌

**验证结果**：
- ✅ 最近两周 `shoplazza_store_hourly` 表中没有 `owner` 为 NULL 的记录
- ✅ 授权负责人的数据存在（每个负责人334条记录）
- ❌ 问题在于查询逻辑：小时粒度使用 JOIN，天粒度直接使用 owner 字段

### 🔍 问题分析

#### 查询逻辑对比

**天粒度查询（正常）**：
```python
# get_daily_data_with_spend_filtered
FROM owner_daily_summary
WHERE date >= DATE(%s) 
  AND date <= DATE(%s)
  AND owner IN ({placeholders})  # ✅ 直接使用 owner 字段
```

**小时粒度查询（有问题）**：
```python
# get_hourly_data_with_spend_filtered
FROM shoplazza_store_hourly s
INNER JOIN store_owner_mapping m ON s.shop_domain = m.shop_domain
WHERE s.time_hour >= %s AND s.time_hour <= %s 
  AND m.owner IN ({placeholders})  # ❌ 使用 JOIN 后的 m.owner
```

#### 问题根源

1. **JOIN字符集冲突**：遇到字符集排序规则冲突错误
   - 错误：`Illegal mix of collations (utf8mb4_0900_ai_ci,IMPLICIT) and (utf8mb4_unicode_ci,IMPLICIT)`
2. **查询逻辑不一致**：天粒度直接使用owner字段，小时粒度使用JOIN
3. **数据验证**：`shoplazza_store_hourly`表中`owner`字段有值，可以直接使用

### 💡 解决方案

**方案：直接使用 owner 字段过滤**

**修改内容**：
1. 移除JOIN：不再使用 `INNER JOIN store_owner_mapping`、`INNER JOIN ad_account_owner_mapping`、`INNER JOIN tt_ad_account_owner_mapping`
2. 直接使用owner字段：使用 `s.owner IN (allowed_owners)`、`f.owner IN (allowed_owners)`、`t.owner IN (allowed_owners)` 过滤
3. 简化参数：不再需要传递映射表的参数

**修改后的SQL**：
```sql
-- 销售数据：直接使用 owner 字段过滤
SELECT 
    s.time_hour,
    SUM(s.total_gmv) AS total_gmv,
    SUM(s.total_orders) AS total_orders,
    MAX(s.total_visitors) AS total_visitors,
    0 AS total_spend
FROM shoplazza_store_hourly s
WHERE s.time_hour >= %s AND s.time_hour <= %s 
  AND s.owner IN ('21', 'Lucky', 'Williams')  -- ✅ 直接使用 owner 字段
GROUP BY s.time_hour
```

### ✅ 修复结果

- ✅ 普通用户可以正常查看小时粒度数据
- ✅ 数据准确，与天粒度数据一致
- ✅ 避免了JOIN字符集冲突问题

## 问题2：小时粒度访客数计算不准确

### 📋 问题描述

**现象**：
- 天粒度访客数：4968 ✅（正确）
- 小时粒度访客数：3398 ❌（错误，少了1570）

**需求**：
- 访客数应该是授权给普通用户的负责人的访客数加起来的数量
- 天粒度和小时粒度的访客数应该一致

### 🔍 问题分析

#### 天粒度访客数计算（正确）

**数据源**：`owner_daily_summary` 表

**查询逻辑**：
```sql
SELECT 
    date,
    SUM(total_visitors) AS total_visitors  -- ✅ 直接累加所有授权负责人的访客数
FROM owner_daily_summary
WHERE date >= DATE(%s) 
  AND date <= DATE(%s)
  AND owner IN ('21', 'Lucky', 'Williams')
GROUP BY date
```

**关键点**：
- `owner_daily_summary` 表的数据是已经按负责人聚合好的
- 每个负责人的 `total_visitors` 是该负责人所有店铺的访客数累加
- 多个负责人的访客数直接 `SUM` 累加即可

#### 小时粒度访客数计算（错误）

**当前逻辑**：
```sql
SELECT 
    t.time_hour,
    MAX(t.total_visitors) AS total_visitors  -- ❌ 只取了某个店铺的最大值
FROM (
    SELECT 
        s.time_hour,
        MAX(s.total_visitors) AS total_visitors  -- 先按 time_hour 分组取 MAX
    FROM shoplazza_store_hourly s
    WHERE s.owner IN ('21', 'Lucky', 'Williams')
    GROUP BY s.time_hour
) t
GROUP BY t.time_hour
```

**问题根源**：
1. **访客数特性**：
   - 访客数是按天去重的累计值，同一天内不同小时是递增的
   - 不同店铺的访客是不同的IP，**应该累加**
2. **当前逻辑的问题**：
   - 只取了某个店铺当天的最大访客数
   - **没有累加所有店铺的访客数**
   - **没有累加所有负责人的访客数**

### 💡 解决方案

**方案：后端预处理访客数**

**正确的逻辑**：
1. 先按 `owner`、`shop_domain`、`DATE(time_hour)` 分组，取每个店铺当天的最大访客数
2. 然后按 `owner`、`DATE(time_hour)` 分组，累加所有店铺的访客数（得到每个负责人当天的总访客数）
3. 最后按 `time_hour` 分组，但同一天所有小时的访客数应该相同（取当天的最大值）
4. 累加所有授权负责人的访客数

**实现方式**：
```python
# 步骤1：查询原始数据（包含 owner, shop_domain, time_hour）
# 步骤2：按 owner, shop_domain, date 分组，取每个店铺当天的最大访客数
shop_daily_visitors = defaultdict(int)
for row in sales_data:
    owner = row['owner']
    shop_domain = row['shop_domain']
    date_key = row['time_hour'].date()
    visitors = row['total_visitors']
    key = (owner, shop_domain, date_key)
    shop_daily_visitors[key] = max(shop_daily_visitors[key], visitors)

# 步骤3：按 owner, date 分组，累加所有店铺的访客数
owner_daily_visitors = defaultdict(int)
for (owner, shop_domain, date_key), max_visitors in shop_daily_visitors.items():
    owner_key = (owner, date_key)
    owner_daily_visitors[owner_key] += max_visitors

# 步骤4：按 time_hour 分组，累加所有授权负责人的访客数
for time_hour in hourly_sales.keys():
    date_key = time_hour.date()
    total_visitors = 0
    for owner in allowed_owners:
        owner_key = (owner, date_key)
        total_visitors += owner_daily_visitors.get(owner_key, 0)
    hourly_sales[time_hour]['total_visitors'] = total_visitors
```

### ✅ 修复结果

- ✅ 小时粒度访客数与天粒度一致（4968）
- ✅ 正确累加所有店铺的访客数
- ✅ 正确累加所有授权负责人的访客数

## 问题3：owner字段NULL记录分析

### 📊 当前情况

**查询结果**：
```sql
SELECT COUNT(*) as null_count
FROM shoplazza_store_hourly
WHERE owner IS NULL OR owner = '';
-- 结果：3288 条记录
```

**验证结果**：
- ✅ 最近两周没有 NULL 记录
- ✅ 说明问题不是 NULL 记录导致的

### 🔍 原因分析

#### 数据同步脚本的 owner 填充逻辑

**写入方法**：
```python
# 每次写入时都会实时查询映射表
owner_sql = "SELECT owner FROM store_owner_mapping WHERE shop_domain = %s"
cursor.execute(owner_sql, (shop_domain,))
owner_result = cursor.fetchone()
if owner_result:
    owner = owner_result['owner']
```

**关键点**：
- 数据同步脚本在写入数据时，会**实时查询** `store_owner_mapping` 表来获取 owner
- 如果映射表中**没有对应的映射关系**，owner 就会是 `None`（即 NULL）

#### 导致 NULL 记录的原因

1. **历史数据**（最可能的原因）：
   - 系统最初运行时，`store_owner_mapping` 表可能还没有完全建立映射关系
   - 数据同步脚本已经开始运行，写入了数据
   - 这些历史数据的 owner 字段就是 NULL
   - 后来虽然建立了映射关系，但**历史数据没有自动回填**

2. **新店铺**：
   - 新店铺添加到 `shoplazza_stores` 表
   - 数据同步脚本开始收集该店铺的数据
   - 但 `store_owner_mapping` 表中还没有添加该店铺的映射关系

3. **增量更新逻辑**：
   - `COALESCE(VALUES(owner), owner)` 的逻辑
   - 如果第一次写入时 owner 是 NULL，后续更新时如果映射表中仍然没有映射关系，owner 仍然是 NULL

### 💡 解决方案

**方案：回填历史数据**

**使用脚本**：`backfill_owner_to_store_hourly.py`

**使用方式**：
```bash
# 回填所有历史数据（需要指定日期范围）
python backfill_owner_to_store_hourly.py --start 2025-01-01 --end 2025-12-22

# 或者回填最近30天
python backfill_owner_to_store_hourly.py --days 30
```

**说明**：
- 脚本只会更新 `owner IS NULL OR owner = ''` 的记录
- 不会覆盖已经存在的 owner 值，所以是安全的

### ⚠️ 注意事项

- **对当前功能无影响**：最近两周没有 NULL 记录，不影响当前功能
- **历史数据**：如果需要查询历史数据，可能需要回填
- **定期维护**：可以设置定时任务，定期运行回填脚本
