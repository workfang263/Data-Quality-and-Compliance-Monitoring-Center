## 📋 项目概述

**项目名称**：Streamlit → Vue + FastAPI 迁移项目

**项目目标**：将现有的 Streamlit 系统迁移到 Vue + FastAPI 架构，实现前后端分离。

**核心原则**：
1. ✅ **完全隔离**：新旧系统互不影响（方案二：复制代码）
2. ✅ **并行运行**：两个系统同时运行，不同端口
3. ✅ **共用数据库**：数据一致，无需迁移
4. ✅ **逐步切换**：功能模块逐步迁移，测试通过后再切换

**技术栈**：
- **后端**：FastAPI 0.104.1 + Uvicorn + PyMySQL + Pydantic
- **前端**：Vue 3.3.13 + TypeScript + Vite + Element Plus + ECharts
- **数据库**：MySQL（共用旧系统数据库）

## 🔐 权限管理系统说明

### 权限类型
1. **负责人权限**：控制普通用户可以查看哪些负责人的数据
2. **查看看板总数据权限**（`can_view_dashboard`）：控制普通用户是否可以查看看板总数据（只显示被授权负责人的聚合总数据）
3. **编辑映射权限**（`can_edit_mappings`）：控制普通用户是否可以编辑映射

### 权限控制效果
- **映射编辑页面**：没有权限时显示提示，禁用所有编辑功能
- **看板页面**：没有权限时显示提示，隐藏总数据；有权限时显示被授权负责人的聚合总数据
- **负责人汇总页面**：只显示被授权的负责人
- **管理员**：自动拥有所有权限

- 新系统有自己的 `database_new.py` 和 `config_new.py`

- 两个系统可以并行运行

### 5. 数据准确性
- ⚠️ **旧系统的数据是准确的**，新系统应该完全对齐旧系统的逻辑
- ⚠️ **访客数是累计值**，同一天内不同小时是递增的（00:00 ≤ 01:00 ≤ ... ≤ 23:00）
- ⚠️ **小时粒度访客数计算**：按天分组取最大值后累加
- ⚠️ **天粒度访客数计算**：直接累加

## 🔑 关键修复记录（最新）

### 权限系统数据准确性修复（2025-12-22）

#### 问题1：普通用户看板小时粒度数据缺失
**问题**：普通用户test被授予了三个负责人权限（21、Lucky、Williams）和"允许查看看板总数据"权限，但看板页面小时粒度数据不显示，天粒度数据正常显示。

**原因**：
- JOIN字符集冲突：`get_hourly_data_with_spend_filtered`方法使用`INNER JOIN store_owner_mapping`过滤，遇到字符集排序规则冲突
- 查询逻辑不一致：天粒度查询直接使用`owner IN (allowed_owners)`过滤，小时粒度查询使用JOIN映射表

**解决方案**：
- 移除JOIN：直接使用`owner`字段过滤，避免字符集冲突
- 与天粒度一致：保持查询逻辑一致性

**相关文件**：
- `backend/app/services/database_new.py` - `get_hourly_data_with_spend_filtered`方法

---

#### 问题2：小时粒度访客数计算不准确（权限系统）
**问题**：普通用户test的天粒度访客数是4968（正确），但小时粒度访客数是3398（错误，少了1570）。

**原因**：
- 访客数累加逻辑错误：只取了某个店铺当天的最大访客数，没有累加所有店铺的访客数
- 不同店铺的访客是不同的IP，应该累加

**解决方案**：
- 后端预处理：先按`owner`、`shop_domain`、`DATE(time_hour)`分组，取每个店铺当天的最大访客数
- 然后按`owner`、`DATE(time_hour)`分组，累加所有店铺的访客数
- 最后累加所有授权负责人的访客数

**相关文件**：
- `backend/app/services/database_new.py` - `get_hourly_data_with_spend_filtered`方法

---

### 数据准确性修复（2025-12-19）
**问题**：新系统与旧系统使用同一个数据库，但访客数不一致

**解决方案**：
1. **完全对齐旧系统逻辑**：
   - 小时粒度：使用`shoplazza_overview_hourly`汇总表，不加映射表过滤
   - 天粒度：使用`owner_daily_summary`表，不加映射表过滤
2. **后端数据预处理**：小时粒度数据在返回前，先按天分组取最大访客数
3. **前端访客数计算**：小时粒度按天分组取最大值后累加，天粒度直接累加

**关键原则**：
- ⚠️ **使用MAX而不是SUM**：因为访客数是累计值，不应该累加，应该取最大值
- ⚠️ **避免时区转换**：解析ISO日期字符串时，直接提取日期部分
- ⚠️ **不同店铺应该累加**：不同店铺的访客是不同的IP，应该累加
