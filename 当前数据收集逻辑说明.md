# 当前数据收集逻辑说明

## 📋 总体流程

当前系统使用 **5分钟实时同步** 机制，通过 Windows 任务计划程序每5分钟执行一次 `run_sync.bat`。

---

## 🔄 执行流程

### 1. 触发机制

- **执行方式**：Windows 任务计划程序，每5分钟自动执行
- **执行脚本**：`run_sync.bat` → `python data_sync.py --realtime`
- **执行频率**：每天 288 次（24小时 × 60分钟 ÷ 5分钟）

### 2. 主要逻辑（`sync_realtime_data_five_minutes()` 函数）

```
开始执行
  ↓
检查是否是00:00:00（跨天处理）
  ↓ 是
  执行 sync_yesterday_final_data() 收集昨天的最终数据
  返回
  ↓ 否
计算当前5分钟段和最近完成的段
  ↓
从 sync_status 表获取最后同步时间
  ↓
根据最后同步时间确定要收集的时间段
  ↓
并行收集所有店铺的数据（35个店铺）
  ↓
按小时汇总数据
  ↓
写入数据库（累加式）
  ↓
更新 sync_status 表
```

## ⏰ 时间段计算逻辑

### 5分钟段划分

一天的时间被划分为多个5分钟段：
- `00:00:00` - `00:04:59.999999`（第1个段）
- `00:05:00` - `00:09:59.999999`（第2个段）
- `00:10:00` - `00:14:59.999999`（第3个段）
- ...
- `23:55:00` - `23:59:59.999999`（第288个段）

### 计算逻辑

**步骤1：计算当前正在进行的段**

```python
current_minute = now.minute  # 例如：17:29 → 29
current_segment_start_minute = (current_minute // 5) * 5  # 29 → 25
current_segment_start = now.replace(minute=25, second=0, microsecond=0)  # 17:25:00
current_segment_end = current_segment_start + 5分钟 - 1微秒  # 17:29:59.999999
```

**步骤2：判断当前段是否已完成**

- 如果 `当前时间 >= current_segment_end`：当前段已完成，收集当前段
- 如果 `当前时间 < current_segment_end`：当前段未完成，收集前一个段

**示例：**
- 当前时间：`17:29:30` → 当前段已完成（17:25:00 - 17:29:59），收集当前段
- 当前时间：`17:27:00` → 当前段未完成（17:25:00 - 17:29:59），收集前一个段（17:20:00 - 17:24:59）

## 📊 数据收集逻辑

### 1. 跨天处理（00:00:00执行时）

- **触发条件**：执行时间在 `00:00:00` - `00:00:09` 之间
- **处理方式**：调用 `sync_yesterday_final_data()` 收集昨天的最终数据
- **完成后**：返回，不执行正常同步流程

### 2. 同步状态检查

**从 `sync_status` 表获取最后同步时间：**

```sql
SELECT last_sync_end_time, last_sync_date, last_visitor_cumulative
FROM sync_status
WHERE sync_type = 'five_minute_realtime'
```

**同步状态格式：**
- `last_sync_end_time`：最后同步段的结束时间（例如：`17:24:59.999999`）
- `last_sync_date`：最后同步日期（例如：`2025-12-30`）
- `last_visitor_cumulative`：最后累计访客数

### 3. 确定要收集的时间段

根据 `last_sync_end_time` 和当前时间，确定要收集的时间段：

#### 情况1：新的一天（最后同步时间不是今天）

- **处理方式**：只收集最近1小时的数据（避免一次性收集太多）
- **开始时间**：`max(今天00:00:00, 1小时前)`
- **结束时间**：最近完成的5分钟段的结束时间
- **提示**：建议运行 `python fill_today_data.py` 补全今天的所有数据

#### 情况2：最后同步时间过旧（超过30分钟）

- **处理方式**：只收集最近1小时的数据（避免一次性收集太多）
- **开始时间**：`max(1小时前, 上次同步后的下一个段)`
- **结束时间**：最近完成的5分钟段的结束时间
- **提示**：建议运行 `python fill_missing_data.py` 补全缺失数据

#### 情况3：正常同步（时间差不超过30分钟）

**判断逻辑：**
- 如果当前段已完成，且当前段未收集 → 收集当前段
- 如果当前段未完成，且最近完成的段未收集 → 收集最近完成的段
- 如果最近完成的段已收集 → 返回，不收集

**关键判断：**
```python
if recent_5min_end <= last_sync_end_time:
    # 最近完成的段已收集，不收集
    return
else:
    # 最近完成的段未收集，收集它
    start_time = recent_5min_start
    end_time = recent_5min_end
```

#### 情况4：时间范围过大（超过1小时）

- **处理方式**：调整为只收集最近1小时的数据
- **提示**：建议运行 `python fill_missing_data.py` 补全缺失数据

## 🏪 店铺数据收集（`sync_store_data_for_five_minutes()`）

### 并行收集所有店铺

- **并发数**：10个线程并行处理（`max_workers=10`）
- **店铺数量**：35个启用店铺
- **收集方式**：每个店铺独立调用 API

### 单个店铺的数据收集流程

```
调用 sync_store_data_for_five_minutes(店铺域名, Access Token, 开始时间, 结束时间)
  ↓
1. 调用订单API（按支付时间查询）
   - API: GET /openapi/2022-01/orders
   - 参数: placed_at_min, placed_at_max (ISO 8601格式)
   - 过滤: 礼品卡订单、COD订单
   - 统计: 订单数、销售额（实际支付金额）
  ↓
2. 调用数据分析API（获取访客数）
   - API: GET /openapi/2022-01/data/analysis
   - 参数: begin_time, end_time (Unix时间戳)
   - 返回: 访客数（当天累计，按天去重）
  ↓
3. 返回结果
   {
     'success': True/False,
     'sales': 销售额（float）,
     'orders': 订单数（int）,
     'visitors': 访客数（int）,
     'error': 错误信息（如果有）
   }
```

### 数据过滤规则

1. **礼品卡订单**：自动过滤，不统计
2. **COD订单**：自动过滤，不统计
3. **价格解析失败**：记录日志，不统计（订单数和销售额）
4. **API调用失败**：记录错误，该店铺数据标记为失败

## 📈 数据汇总和写入

### 1. 汇总所有店铺的数据

```python
total_sales = 所有店铺的销售额之和
total_orders = 所有店铺的订单数之和
total_visitors = 所有店铺的访客数之和
```

**注意**：访客数虽然每个店铺是按天去重的，但多个店铺之间应该累加（不同店铺的访客是不同的IP）。

### 2. 确定数据属于哪个小时

```python
time_hour = start_time.replace(minute=0, second=0, microsecond=0)
```

**示例：**
- 收集时间段：`17:20:00` - `17:24:59` → 属于 `17:00:00` 这个小时

### 3. 写入单店铺明细表（`shoplazza_store_hourly`）

**对每个店铺：**
- 使用 `insert_or_update_store_hourly_incremental()` 函数
- **写入模式**：**累加式**（`total_gmv = total_gmv + VALUES(total_gmv)`）
- **访客数**：直接覆盖（`total_visitors = VALUES(total_visitors)`）

```sql
INSERT INTO shoplazza_store_hourly
(shop_domain, owner, time_hour, total_gmv, total_orders, total_visitors, avg_order_value)
VALUES (%s, %s, %s, %s, %s, %s, %s)
ON DUPLICATE KEY UPDATE
    total_gmv = total_gmv + VALUES(total_gmv),        -- 累加
    total_orders = total_orders + VALUES(total_orders),  -- 累加
    total_visitors = VALUES(total_visitors),          -- 覆盖
    avg_order_value = (total_gmv + VALUES(total_gmv)) / (total_orders + VALUES(total_orders))
```

### 4. 写入汇总表（`shoplazza_overview_hourly`）

- 使用 `insert_or_update_hourly_data_incremental()` 函数
- **写入模式**：**累加式**（`total_gmv = total_gmv + VALUES(total_gmv)`）
- **访客数**：直接覆盖（`total_visitors = VALUES(total_visitors)`）

```sql
INSERT INTO shoplazza_overview_hourly
(time_hour, total_gmv, total_orders, total_visitors, avg_order_value)
VALUES (%s, %s, %s, %s, %s)
ON DUPLICATE KEY UPDATE
    total_gmv = total_gmv + VALUES(total_gmv),        -- 累加
    total_orders = total_orders + VALUES(total_orders),  -- 累加
    total_visitors = VALUES(total_visitors),          -- 覆盖
    avg_order_value = CASE
        WHEN (total_orders + VALUES(total_orders)) > 0
        THEN (total_gmv + VALUES(total_gmv)) / (total_orders + VALUES(total_orders))
        ELSE 0.0
    END
```

**关键点：**
- ✅ **累加式写入**：订单数和销售额是累加的（`total_gmv = total_gmv + VALUES(total_gmv)`）
- ✅ **覆盖式写入**：访客数是覆盖的（`total_visitors = VALUES(total_visitors)`）
- ⚠️ **数据翻倍风险**：如果两个进程同时执行，累加写入会导致数据翻倍

### 5. 更新同步状态（`sync_status` 表）

```python
db.update_sync_status(
    sync_type='five_minute_realtime',
    last_sync_end_time=end_time,  # 本次收集段的结束时间
    last_sync_date=today_start.date(),  # 今天日期
    last_visitor_cumulative=total_visitors  # 累计访客数
)
```

**更新逻辑：**
- `last_sync_end_time`：存储**已收集段的结束时间**（例如：`17:24:59.999999`）
- `last_sync_date`：最后同步日期
- `last_visitor_cumulative`：最后累计访客数

## 📝 示例流程

### 示例1：正常同步（17:25:00执行）

**当前时间**：`2025-12-30 17:25:30`

1. **计算时间段**：
   - 当前段：`17:25:00` - `17:29:59`
   - 当前段已完成 → 收集当前段：`17:25:00` - `17:29:59`

2. **获取最后同步时间**：`17:24:59.999999`（上次收集了 17:20:00 - 17:24:59）

3. **判断**：`17:29:59.999999 > 17:24:59.999999` → 需要收集

4. **收集数据**：
   - 并行收集35个店铺的数据（17:25:00 - 17:29:59）
   - 假设结果：订单 10 单，销售额 $500，访客 100

5. **写入数据库**：
   - 汇总表：`17:00:00` 小时的数据累加（订单+10，销售额+$500，访客=100）
   - 单店铺明细表：每个店铺的数据累加

6. **更新同步状态**：`last_sync_end_time = 17:29:59.999999`

### 示例2：进程重叠（竞态条件）

**场景**：第一个进程执行时间过长（6-7分钟），第二个进程在第一个进程完成前启动

**时间线：**
- `T=17:25:00`：第一个进程启动，收集 `17:25:00` - `17:29:59`
- `T=17:30:00`：第二个进程启动（第一个进程仍在运行）
- `T=17:32:00`：第一个进程完成，写入数据库（订单+10，销售额+$500）
- `T=17:33:00`：第二个进程完成，也写入数据库（订单+10，销售额+$500）

**结果**：
- 数据库中的订单数：10 + 10 = 20（应该是10）
- 数据库中的销售额：$500 + $500 = $1000（应该是$500）
- **数据翻倍！** ❌

**原因**：
- 两个进程都查询到相同的已有数据（例如：订单 100，销售额 $5000）
- 第一个进程：`100 + 10 = 110`，写入 110
- 第二个进程：`100 + 10 = 110`（在第一个进程写入前查询），写入 110
- 但由于是累加写入（`total_gmv = total_gmv + VALUES(total_gmv)`），最终结果可能是 120（如果第二个进程在第一个进程之后写入）

## ⚠️ 当前问题

### 1. 数据翻倍风险

- **原因**：累加式写入 + 进程重叠
- **表现**：数据库数据是API数据的2倍（或更多倍）
- **影响**：数据不准确，影响业务决策

### 2. 系统负载较高

- **执行频率**：每5分钟执行一次（每天288次）
- **API调用**：35个店铺 × 288次 = 约10,080次/天
- **数据库写入**：约10,080次/天

---

## 🔧 优化建议

### 方案1：改为10分钟间隔 + 覆盖式写入（推荐）

- ✅ **降低执行频率**：从5分钟改为10分钟（减少50%）
- ✅ **改为覆盖式写入**：即使进程重叠也不会翻倍
- ✅ **双重保障**：从根本上解决数据翻倍问题

详细方案请参考：`综合优化方案：10分钟间隔+覆盖式写入.md`
